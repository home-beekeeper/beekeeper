## 🔴 BLOCKER — FRB-05: local overlay is written but never enforced at check time

**The moat's local-feedback loop does not close.** Phase 24 writes a confirmed-malicious
package into `local-overlay.idx` (`catalogs_daemon.go:169`), and the catalog layer can query
it as a 4th source (`multi.go:101-106`, `CatalogSource="local-overlay"`). But **no production
enforcement path constructs the MultiIndex with the overlay**:

| Surface | File:line | Constructor used |
|---------|-----------|------------------|
| `beekeeper check` (hook hot path) | `internal/check/handler.go:236` | `NewMultiIndex` (no overlay) |
| MCP gateway | `internal/gateway/gateway.go:141` | `NewMultiIndex` |
| scan | `internal/scan/scanner.go:368` | `NewMultiIndex` |
| watch crossref | `internal/watch/crossref.go:190` | `NewMultiIndex` |
| watch handler | `internal/watch/handler.go:122` | `NewMultiIndex` |
| catalogs daemon | `cmd/beekeeper/catalogs_daemon.go:113` | `NewMultiIndex` |

`NewMultiIndexWithOverlay` is invoked **only from test files**. So a confirmed-malicious overlay
entry contributes **zero matches** to `policy.Evaluate` on the live path — the package passes
with only the original Bumblebee/OSV/Socket matches, exactly as if the overlay did not exist.

**Why the green suite missed it:** every overlay test
(`TestRunCatalogsSyncFirstResponder` step 7, `TestCorpusMoatLoopE2E`,
`local_overlay_test.go`) exercises `NewMultiIndexWithOverlay` **directly**, never through a live
`RunCheck`. The production wiring gap is invisible to the test suite — the same false-confidence
pattern that bit Phase 25's launch gates (a test that asserts the component, not the path).

**Fix path:** in `internal/check/handler.go` (after `bbIdx`), derive
`filepath.Join(cacheDir, "local-overlay.idx")` and call
`catalog.NewMultiIndexWithOverlay(bbIdx, osvAdapter, socketAdapter, overlayPath)`; close
`multiIdx.Overlay` when present. Decide which other surfaces (gateway, scan, watch) also need
the overlay. Add a **RunCheck-level** regression test: seed an overlay entry, run `RunCheck`
on a matching install, assert the overlay match pushes a 1-source warn to a 2-source enforce.

## ⚠ WARNING — STORE-02: multi-sink fan-out seam unused

`audit.NewMultiSinkWithCorpus` (the Phase-23 designed seam to fan every `audit.Write` into the
corpus store) has **no production call site**; `handler.go` writes corpus records via
`writeCorpusRecordDirect` / `AppendCorpusRecordLine` instead. Records still reach the corpus
file (verified by Phase-23 store tests), so this is a design deviation rather than a functional
break — but the spec'd architecture is not what runs. Non-blocking.

- **Config merge (Phase-24 fix):** `mergeCorpus`/`mergeAutoQuarantine` called in both `merge()`
