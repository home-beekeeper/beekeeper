## v1.5.0 Requirements — all satisfied

### Install Posture — Layer 1 enforcement (IPST)
- [x] **IPST-01**: Default install posture (release-age <24h warn, lifecycle-script warn, git/remote-URL warn) enforced at the pre-exec hook via the existing engine, on by default. — Phase 27 (`posture_adapter` → `handler` most-restrictive merge)
- [x] **IPST-02**: Release-age enforcement catches a version published under the window (default 24h) and surfaces the reason. — Phase 27 (`policy.EvaluateReleaseAge` + adapter)
- [x] **IPST-03**: Lifecycle-script posture warns on an unapproved install script, raisable to block per rule. — Phase 27/29
- [x] **IPST-04**: Git/remote-URL dependency installs detected + flagged/warned with reason (NEW rule). — Phase 26 (`pkgparse.RemoteSource` + `policy.EvaluateRemoteSource`)
- [x] **IPST-05**: Posture rules are inputs to the existing pure policy engine, not a parallel mechanism. — Phase 26 (pure evaluators, I/O-free, consumed by the check handler)
- [x] **IPST-06**: Sentry observes + audits installs it sees (agent or human-run), labeled detection not prevention. — Phase 27 (SENTRY-009 `install_observe.go` → `sentry_install_observed`)

### Install Posture — boundaries & honesty (IPBND)
- [x] **IPBND-01**: The enforcement-boundary statement appears wherever install posture is described (code, help, docs, web copy). — Phases 27 + 30 (`posture.BoundaryStatement` propagated)

### Install Posture — Layer 2 read-only view (IPVIEW)
- [x] **IPVIEW-01**: `beekeeper posture` (CLI + TUI) reads each detected pm's config read-only and displays it side by side with Beekeeper's enforced posture, naming gaps. Machine-wide. — Phase 28 (shared `view.BuildComparison`)
- [x] **IPVIEW-02**: The view writes no pm config file — read-only guarantee asserted by test. — Phase 28 (`TestPostureCmd_ReadOnlyGuarantee`, sha256 byte-for-byte)

### Install Posture — Layer 3 scoped override + per-rule severity (IPOVR)
- [x] **IPOVR-01**: A posture-rule decision (warn OR block) offers graduated scoped responses (allow once / allow always with recorded reason / block). — Phase 29
- [x] **IPOVR-02**: Each override is a distinct `posture_override` audit record; allow-always persists as a POSTURE-SCOPED standing exception (consulted only by the posture adapter), NOT the general `package_allowlist` (T-09-31 — never whitelists a package past malware detection). — Phase 29 (proven by `TestRunCheckPostureAllowAlwaysDoesNotBypassCatalogBlock`)
- [x] **IPOVR-03**: A user can raise an individual posture rule warn→block via layered config; untrusted (project) layers may only TIGHTEN, never loosen; the unknown path stays fail-soft warn. — Phase 29 (`mergePostureUntrusted` + `posturizeWithAction`). *(Maintainer-directed at Gate 1; pulled part of the PRD roadmap into v1.0.)*
