| RenderDeny → harness | The runtime deny shape (flat vs decision vs dual vs exit-2-only) must match the harness family or the harness fails open |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-13 | Tampering (wrong deny family) | Copilot flat vs nested | mitigate | RenderDeny(HarnessCopilot) emits the FLAT permissionDecision form (asserted in Plan 01 TestRenderDeny); the installer wires --hook copilot so the runtime family matches |
| T-10-14 | Tampering (Windsurf fail-open) | Windsurf exit-2-only + OS key | mitigate | Windsurf fails OPEN on non-2 exit; RenderDeny(HarnessWindsurf) emits exit 2 (Plan 01); the installer wires the powershell key on Windows (the dev box) so the hook actually executes on the primary platform; the GOOS-branch test asserts the right key |
| T-10-15 | Tampering (Antigravity ambiguous deny) | dual-defensive family | mitigate | RenderDeny(HarnessAntigravity) emits BOTH decision:"deny" and permissionDecision:"deny"+denyReason (MED-confidence docs); the installer wires --hook antigravity so whichever field the harness reads, a deny is present |
| T-10-16 | Spoofing (clobber user hooks) | all four installers | mitigate | settings.json installers reuse the merge trinity; typed-struct installers use containsXxxHookByCommand + filtered-append; preserves_existing_hooks subtests assert survival |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No package installs, no new dependencies (stdlib + existing internal/editorinit only) |
