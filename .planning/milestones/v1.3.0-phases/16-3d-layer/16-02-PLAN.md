<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server HTML → client hydration | The ssr:false boundary; 3D code must never cross into server-rendered output |
| Browser GPU → WebGL context | Canvas allocates a WebGL context; flooding is the DoS risk |
| pointer input → PresentationControls | Untrusted pointer drag, bounded to polar/azimuth ranges |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-03 | Denial of Service | `<Canvas>` WebGL context | mitigate | Single-canvas rule (exactly one `<Canvas>` page-wide; ambient accents are children of it or CSS, never a 2nd canvas); R3F auto-disposes on unmount; GFX-02 test asserts count <= 1 |
| T-16-04 | Tampering | server HTML cleanliness | mitigate | All three/R3F imports behind use-client + `dynamic(ssr:false)`; GFX-01 grep asserts no `<canvas`/`@react-three`/`WebGLRenderingContext` in out/index.html |
| T-16-05 | Information disclosure | canvas fingerprinting | accept | `aria-hidden` canvas + `failIfMajorPerformanceCaveat:true` reduce surface; full prevention infeasible for a marketing 3D layer — acceptable residual (RESEARCH.md Security Domain) |
| T-16-06 | Tampering | hero SVG render path | mitigate | SVG served via `<img src>` only — never `dangerouslySetInnerHTML`; static `public/` asset |
</threat_model>
