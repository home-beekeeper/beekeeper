**Breaking-change accuracy (CRITICAL — the change is real, document it correctly):**
The exit-1 → exit-2 change is the v1.3.0 seed fix shipped in Phase 10 (2026-06-05). The accurate, verified facts (from MILESTONES.md context + the Phase-10 seed history):
- BEFORE (pre-v1.3.0): `beekeeper check` exited `1` and emitted its own decision JSON on a block. Agent harnesses require exit `2` (Claude Code) or a per-harness deny JSON (e.g. Hermes `{"action":"block"}`) to actually deny a tool call. The hook FIRED and AUDITED but the tool ran anyway — a silent enforcement hole on every harness.
- AFTER (v1.3.0): `beekeeper check --hook <harness>` emits the correct per-harness deny contract — exit `2` + `hookSpecificOutput` for Claude Code; `{"action":"block"}` with a guaranteed-non-empty message for Hermes; unknown harness fails closed (exit 2). A `--hook` leak that emitted the raw `{"Allow":false}` Decision JSON (which would silently ALLOW on Hermes — fail-open) was also fixed.
- MIGRATION: users must (a) upgrade to v1.3.0, and (b) re-run their harness hook installer (`beekeeper hooks install <harness>` / re-register the PreToolUse runner) so the hook command uses `--hook <harness>` and the deny contract is wired. Claude Code loads hooks at session start — a session restart is required for the new hook to take effect. Without this, blocks were silently ignored.
The migration note in the callout MUST state these accurately. Do NOT overstate (e.g. do not claim all 15 harnesses are live-verified — only Claude Code is live-verified; the rest are contract-shape tested).
</design_decisions>
