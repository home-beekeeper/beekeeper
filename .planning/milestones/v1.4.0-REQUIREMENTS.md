**Defined:** 2026-06-13
**Core Value:** A hijacked or off-task agent cannot successfully act on the developer's machine without Beekeeper deciding to permit it.
**Milestone goal:** Capture the moat — incidents with confirmed ground-truth outcomes attached — from the first run on a single, offline machine; wire confirmed adjudications into First Responder; and freeze the push-envelope wire format without standing up any transport.
**Scope source:** `beekeeper-corpus-milestone-prd.md` §3 ("v1 scope — build now") only. PRD §6 (v1.1–1.9 org push) and §7 (v2.0 community feed) are future milestones.

## v1 Requirements

Requirements for this milestone. Each maps to exactly one phase (22–25).

### Schema (SCHEMA) — Phase 22

- [x] **SCHEMA-01**: A four-layer event record (behavior / decision / outcome / context) is defined as a typed Go schema; `CorpusRecord` embeds the existing `AuditRecord` (which already carries behavior + decision) and adds the outcome + context layers; `source_surface` is an additive field on the base record with six branch values (`hook | mcp_gateway | shim | file_watcher | sentry | scan`). Conditional fields populate only for the relevant surface; the outcome-layer fields are present as `unresolved` placeholders from the very first corpus write (non-retrofittable).
- [x] **SCHEMA-02**: `cluster_id` binds correlated events into one incident. Non-agent surfaces (`file_watcher`, `sentry`, `scan`) adjudicate the cluster keyed by `cluster_id`; agent-mediated surfaces (`hook`, `mcp_gateway`, `shim`) adjudicate per event. For the `scan` surface the unit is per-package-hit with a STABLE key `cluster_id = hash(package_or_extension_id + version + repo_fingerprint)`, so re-scans of the same flagged version map to the same (already-adjudicated) incident rather than minting duplicates (OQ-2 locked).
- [x] **SCHEMA-03**: The push-envelope wire format is frozen as a typed schema: `signature{package_or_extension_id, version, behavior_signature_hash, iocs}`, `true_label`, `confidence_tier`, `source_count`, `scope`, `action_hint`, and a `signing{issuer, signature, issued_at, nonce}` block (zero-value in v1).
- [x] **SCHEMA-04**: `action_hint` is a typed constant whose only pushable value is `watch_and_block`; `auto_purge` is unrepresentable in a pushable envelope (enforced as a compile-time guard, not a runtime string check).
- [x] **SCHEMA-05**: The `behavior_signature_hash` input definition is frozen in this phase (hash over `action_type` + normalized `target_resource` + normalized `network_destination`) and is stable across schema versions; a `ruleset_version` field is recorded on every record so later schema evolution is detectable.
- [x] **SCHEMA-06**: Schema-lock evaluator gate — every field in the Nx Console worked trace maps to a schema field with no gaps, and the envelope can represent a `watch_and_block` push carrying `confidence_tier` and `source_count`.

### Scope Tagging (SCOPE) — Phase 22

- [x] **SCOPE-01**: Every record carries a `scope` field from birth (`org_only | community_shareable`); the Go zero-value resolves to `org_only`.
- [x] **SCOPE-02**: Promotion to `community_shareable` is explicit and gated by the anonymization step (delivered in v2.0); in v1 the promotion code path always returns an error and is the only path that changes `scope`. Promotion is never automatic.

### Adjudication (ADJ) — Phase 23

- [x] **ADJ-01**: The adjudication engine assigns the outcome layer asynchronously, off the hot path — it never blocks the synchronous fail-closed hook handler, and any pre-exit flush has a bounded deadline. The engine lives in a new impure package and consumes the existing pure corroboration logic; `internal/policy` stays I/O-free. The AUTOMATIC adjudication loop (`catalog_confirmation`, `downstream_clean`) runs as a background goroutine in the long-lived `catalogs daemon`, with a bounded batch pass at the start of each `catalogs sync` as the no-daemon fallback; it never runs in the `beekeeper check` hot path. Operator-driven sources (`forensic_review`, `breach_confirmation`, `user_override`, `benign_explained`) are synchronous CLI/TUI writes regardless (OQ-3 locked).
- [x] **ADJ-02**: `true_label` ∈ {`malicious`, `benign`, `policy_correct`, `unresolved`}; the initial state is always `unresolved`. `policy_correct` distinguishes a correct deterministic policy hit (where no attack occurred) from `benign`.
- [x] **ADJ-03**: `adjudication_source` ∈ {`catalog_confirmation`, `forensic_review`, `breach_confirmation`, `user_override`, `downstream_clean`, `benign_explained`}, assigned per the documented confidence mapping (forensic/breach = high; catalog/benign_explained = medium; downstream_clean/user_override = weak).
- [x] **ADJ-04**: `source_count` is derived from the count of DISTINCT corroborating sources (deduplicated by source name), never from event count.
- [x] **ADJ-05**: `confidence_tier` = `watch` when `source_count == 1` and `enforce` when `source_count >= 2`; `enforce` requires `source_count >= 2` in all cases (a single-source critical escalation is never tagged `enforce`).
- [x] **ADJ-06**: `was_correct` is derived from `true_label` vs the recorded `verdict` (`policy_correct` counts as `was_correct = true`); `resolved_at` (RFC3339) is set when a label leaves `unresolved` and is absent while still unresolved.
- [x] **ADJ-07**: Corrections are append-only superseding records that reference the prior event/cluster — never in-place mutation. `downstream_clean` labels an allow `benign` only after a **30-day (configurable) observation window** with no correlated follow-on incident, correlation scoped per-machine to the package/extension identity; it remains the weakest benign signal (never enforce) (OQ-1 locked).

### Corpus Store (STORE) — Phase 23

- [x] **STORE-01**: An append-only local corpus store (`corpus.ndjson`) extends the audit-log sink model — implemented as an `audit.Sink` added to the multi-sink, so all six source surfaces gain corpus writing with no per-surface code change.
- [x] **STORE-02**: Every corpus write routes through `RedactRecord` + `DefaultRedactPatterns` before persistence (the F-1 first-responder security-review lesson); the store never opens its own un-redacted write path.
- [x] **STORE-03**: Owner-only permissions (0600 / Windows owner-DACL), identical to the audit log; no off-machine transmission of corpus data in v1 (remote sinks are not wired for corpus).
- [x] **STORE-04**: Records are persisted in push-envelope shape from the first write, so later transport is wiring rather than a migration.
- [x] **STORE-05**: `repo_fingerprint` and `fleet_node_id` are non-reversible — `HMAC-SHA256` over the canonical id with a per-install secret generated on first run, stored owner-only, and never transmitted (the v2.0 anonymization step becomes a salt swap, not a redesign).

### Push Envelope (ENV) — Phase 23

- [x] **ENV-01**: Local records are emitted in the frozen push-envelope shape; NO transport is stood up in v1.
- [x] **ENV-02**: `BuildPushEnvelope` returns an error for any purge-class intent; `auto_purge` is never emitted; `confidence_tier` and `source_count` are frozen at emission time (consumers never re-derive them).
- [x] **ENV-03**: A fuzz / property test asserts that no constructed envelope can escape with a non-allowlisted `action_hint`.

### First Responder Binding (FRB) — Phase 24

- [x] **FRB-01**: A confirmed-malicious adjudication arms a TUI quarantine card for any matching install present locally, via the existing crossref / quarantine seam (no new IPC).
- [x] **FRB-02**: Purge stays a local human-confirmed OR org-policy-gated action — never automatic, and never a fleet-pushable action in any version.
- [x] **FRB-03**: Restore remains available for reversibility after adjudication; the locked TUI semantic is preserved (red = attacker action, coral = Beekeeper response).
- [x] **FRB-04**: A confirmed-malicious adjudication elevates a DETECTION-ONLY Sentry watch on the associated process subtree, gated at corroboration ≥ threshold (reusing the F-4-hardened `TargetList` seam; no kill/isolate).
- [x] **FRB-05**: A confirmed-malicious adjudication adds a local-only catalog overlay entry (owner-only; does not mutate the synced catalog; survives `catalogs sync`) so future installs are caught without waiting for catalog lag.

### Launch Readiness (LAUNCH) — Phase 25

- [x] **LAUNCH-01**: End-to-end — the Nx Console incident runs through trace → record → adjudication → signature → local feedback (catalog overlay + Sentry watch), with all four layers populated.
- [x] **LAUNCH-02**: Each of the eight Sentry patterns (SENTRY-001..008) produces a moat-grade record with all four layers populated.
- [x] **LAUNCH-03**: A disconnected / offline machine remains fully protective on the last synced catalog; `beekeeper check` p99 stays sub-100ms with the corpus enabled (benchmark gate confirms the corpus loop is off the hot path).
- [x] **LAUNCH-04**: No corpus data leaves the machine (verified); documentation states local-first and NAMES the residual gaps — SENTRY-008 CI-runner OIDC theft, GitHub API dead-drop exfil, and DNS-tunnel ingested-but-undetected — in `docs/THREAT-MODEL.md`.

## Locked Decisions

Three research-surfaced gray areas, resolved with the maintainer at milestone setup (2026-06-13). These fix the implemented value/shape; they do not change the requirements. The owning phase's `/gsd-discuss-phase` may refine details but should treat these as the agreed starting point.

| ID | Decision | Resolution | Owning phase |
|----|----------|-----------|--------------|
| OQ-1 | `downstream_clean` observation window | **30 days, configurable.** Weakest benign signal — never enforce, supersedable; correlation scoped per-machine to the package/extension identity. Balances delayed supply-chain detonation against label-coverage latency. | Phase 23 (ADJ-07) |
| OQ-2 | `scan` surface `cluster_id` unit | **Per-package-hit, stable key** `cluster_id = hash(package_or_extension_id + version + repo_fingerprint)`. Independently adjudicable, idempotent across re-scans (no duplicate incidents), aligned with one-card-per-install First Responder. | Phase 22 (SCHEMA-02) |
| OQ-3 | Adjudication-engine lifecycle (automatic sources) | **Background goroutine in the existing `catalogs daemon`** + a bounded batch pass on each `catalogs sync` as the no-daemon fallback; never in the `beekeeper check` hot path. Operator-driven sources stay synchronous CLI/TUI writes. No new dedicated daemon. | Phase 23 (ADJ-01) |

## v2 Requirements

Deferred to future milestones (PRD §6, §7). Tracked, not in this roadmap.

### Org Self-Host Aggregation & Push (PRD §6 — v1.1–1.9)

- **ORG-01**: Self-hosted org aggregator endpoint receiving signed signatures from org machines (org runs its own; no maintained cloud on our side)
- **ORG-02**: Push fan-out within an org — a confirmed incident becomes a signed push; receiving machines run a scoped Bumblebee/Pollen scan, raise Sentry watch, block the pull, and arm the quarantine card
- **ORG-03**: Warn-vs-enforce push gating — one source pushes at watch weight; two-source corroboration required before enforce weight; enforce never triggers automatic destructive action
- **ORG-04**: Signed channel — authenticated pub/sub fan-out with per-push signature verification (populates the v1-frozen `signing` block)
- **ORG-05**: Offline fallback — a machine without the push channel falls back to existing pull-sync behavior

### Community Shared Corpus Feed (PRD §7 — v2.0)

- **COMM-01**: Maintained shared community ingestion + fan-out host
- **COMM-02**: Anonymization step promoting `org_only` → `community_shareable` (strip to threat intelligence only; attacker identity + behavior hash + IOCs, never the victim)
- **COMM-03**: Community catalog as a pull source riding the existing `catalogs sync` alongside Bumblebee/OSV/Socket/Pollen
- **COMM-04**: Community push fan-out under the same `watch_and_block` constraint + corroboration gate
- **COMM-05**: Abuse resistance — signed pushes, rate limiting, poisoned-signature resistance

## Out of Scope

Explicitly excluded for v1.4.0. Anti-features carry version scope.

| Feature | Reason |
|---------|--------|
| `auto_purge` as a pushable action | A fleet-wide destructive action triggered by a remote signal is itself an attack surface. Excluded in ALL versions — `watch_and_block` is the only pushable action; purge stays local + human/policy gated. |
| Network transmission of corpus data | v1 is local-only. Records are emitted in envelope shape so v1.1+ transport is wiring. Transmitting pre-anonymization incident data is a privacy violation and risks a breaking wire change. |
| Org aggregator / push fan-out | High-blast-radius channel; built + hardened post-launch across v1.1–1.9 before carrying enforce-weight signals. |
| Community shared feed / maintained host | Requires maintained infra + anonymization + abuse resistance; v2.0. |
| Automatic `community_shareable` promotion | Promotion is always explicit + anonymization-gated (v2.0), never automatic — prevents accidental victim-data sharing. |
| SENTRY-008 CI-runner OIDC theft detection | Tokens stolen from runner process memory on a machine with no editor/host agent; out of host scope. Architectural mitigation only; NAMED in the threat model (LAUNCH-04). |
| GitHub API dead-drop exfil detection | A host tool cannot reliably distinguish a legitimate push from malicious repo creation with stolen secrets. Architectural mitigation only; NAMED in the threat model (LAUNCH-04). |
| Corpus data mutation / retroactive relabel | Append-only is a forensic requirement. Corrections are superseding records referencing the prior event (ADJ-07), never edits/deletes. |
| Weighted fractional corroboration (0.5 source weight) | TM-B-01 from the 2026-06-05 threat audit — a core-semantics change needing a designed change. The binary 1 / 2+ model is intentional and auditable. |
