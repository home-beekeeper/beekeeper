## Why this is gaps_found, not tech_debt

Three of the four blockers make a shipped requirement non-functional at runtime:

- **INT-BLOCK-1** — the entire LlamaFirewall feature (LLMF-02/03/04/05) is dead code at runtime. Mitigating context: LLMF is disabled-by-default, and Phase 6 explicitly deferred the gateway wiring. But `enable` does nothing useful today.
- **INT-BLOCK-2** — `corroboration_threshold` policy rules silently do nothing in live `beekeeper check` while working in `policy test`. This is a correctness divergence introduced by the Phase 9 / 09-06 policy-as-code wiring and is the most important to fix because it makes the headline "policy as code" feature partially lie to the user.
- **INT-BLOCK-3** — the MCP gateway cannot enforce corroboration (PLCY-01) because it only wires Bumblebee. A hijacked agent routed through the gateway gets weaker enforcement than one going through the hook handler.
- **INT-BLOCK-4** — `beekeeper diag` always reports sidecar latency as 0 (CODE-06 partial), coupled to BLOCK-1.

## Verified working (the core spine)

The hook → policy → catalog → audit → TUI spine is wired and functional: `beekeeper check` builds a 3-source `MultiIndex`, evaluates corroboration (1/2/3 → warn/block/quarantine), applies the policy overlay (package_allowlist + sensitive_path), emits an NDJSON audit record, and the TUI reads it live. Self-quarantine guards all enforcement commands. Sentry daemons (Linux/macOS/Windows) talk to the CLI over IPC and write audit records. Extension defense (watch → scan → quarantine) is wired. 14 cross-phase connections verified with file:symbol evidence.

## Recommended gap closure (before tagging v1.0.0)

A single focused closure phase can fix all four blockers + the two warnings — they are localized wiring, not new features:

1. **INT-BLOCK-2 (highest priority, correctness):** in `internal/check/handler.go runCheck`, derive thresholds from the loaded policy files (export `thresholdsFromPolicyFile` or add a `policyloader` helper) and pass them to `policy.Evaluate` instead of `DefaultCorroborationThresholds()`. Fix the false comment. Add a handler test proving a `corroboration_threshold` rule changes a live decision.
2. **INT-BLOCK-3:** in `internal/gateway/gateway.go`, construct OSV + Socket adapters from `Config.CacheDir`/`SocketToken` (mirror `handler.go:194-213`) and pass to `NewMultiIndex`.
3. **INT-BLOCK-1 + INT-WARN-2:** start the supervisor when `config.LlamaFirewall.Enabled` and route `audit-record` → `RunAuditRecordWithLLMF` and the gateway proxy → `ScanProxiedResponse`. (Or, if LLMF stays out-of-scope for v1.0.0, explicitly document it as deferred and update LLMF requirement statuses to reflect "config-only" — a decision for the user.)
4. **INT-BLOCK-4:** have `Supervisor.Scan` also `GlobalLatencyTracker.Record(...)` (or point diag at the running supervisor's tracker).
5. **INT-WARN-1:** apply `policyloader.LoadPolicyDir` + `ApplyPolicyOverlay` in gateway/watch/scan, or document that policy-file overlays are hook-handler-only for v1.0.0.

Bookkeeping (cheap, do alongside): regenerate VERIFICATION.md for phases 1/2/5/6/7 (or accept UAT as the verification of record) and refresh the REQUIREMENTS.md traceability table to mark shipped requirements Complete.
