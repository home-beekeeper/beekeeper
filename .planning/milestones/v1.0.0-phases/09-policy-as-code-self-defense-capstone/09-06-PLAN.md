# Plan 09-06 — Gap closure: live policy enforcement + self-hosted key + percentile fix

<objective>
Close the three gaps surfaced by verification + code review of Phase 9:

1. **CODE-01 live enforcement (verifier Item 1, user-approved):** Declarative policy
   files in `policies/` are currently validated and listable, but only
   `corroboration_threshold` rules affect any decision. `package_allowlist` and
   `sensitive_path` rules are enforced NOWHERE — not even in `policy test`. Implement
   a PURE policy-overlay evaluator and wire it into BOTH `policy test` and live
   `beekeeper check` so policies are genuinely "loaded by the policy engine"
   (REQUIREMENTS.md CODE-01). `internal/policy` must remain a pure library — the
   overlay lives in `internal/policyloader`.

2. **CR-01 self-hosted beekeeper-self key:** `cfg.SelfCatalog.PubKey` is parsed into
   config but never wired through to `CheckSelfCatalog`, so a user's self-hosted feed
   signed with their own key can't be verified (the override field is unexported).
   Export it and wire it through `enforceSelfQuarantine`.

3. **WR-03 percentile accuracy:** `LatencyTracker.P95()/P99()` use
   `int(float64(n)*p)`, which is ~1 rank high and returns the maximum for small n.
   Correct to nearest-rank without breaking existing Phase 6 latency tests.
</objective>

<design>
## Policy overlay semantics (CODE-01) — PINNED

Add a PURE function in `internal/policyloader/enforce.go`:

```
func ApplyPolicyOverlay(files []PolicyFile, tc policy.ToolCall, base policy.Decision) policy.Decision
```

It evaluates rule types decidable from the ToolCall alone and combines with `base`
(the engine decision) by **most-restrictive-wins, with an explicit allowlist escape hatch**:

- `package_allowlist` (action allow|warn|block, packages[], ecosystem/ecosystems):
  matches when the tool call is a package operation whose package ∈ packages AND the
  ecosystem matches (empty ecosystem = any). Action applied per the combination rule.
- `sensitive_path` (action warn|block, path_patterns[]): matches when the tool call's
  target path matches any pattern (use filepath.Match / prefix per existing path policy
  conventions — read internal/policy/path.go for the matching idiom). Action applied.
- `corroboration_threshold`: NOT handled here — it already flows through
  `thresholdsFromPolicyFile` into `policy.Evaluate`. Skip it in the overlay.
- `release_age` and `lifecycle_script_allowlist`: require package-age / lifecycle data
  NOT present in a pure ToolCall. They are NOT enforced by the overlay in v1; the
  engine's built-in release-age/lifecycle policies (catalog/config-driven) remain the
  enforcement path. Document this limitation in a code comment AND in docs/THREAT-MODEL.md.

**Combination (final decision):**
1. If any matched rule action is `block` → block (reason cites rule ID + file name).
2. Else if an `allow` package_allowlist rule matches the EXACT package AND base is
   warn or block → downgrade to allow (explicit, version-controlled user-trust
   override; this is the point of policy-as-code allowlisting). Reason cites the rule.
3. Else final = most-restrictive of {base, any matched `warn`}.
4. Else → base unchanged.

**Security note (must be documented in docs/THREAT-MODEL.md):** a `package_allowlist`
`allow` rule can override a catalog-corroborated block for that exact package. This is
intentional — the user's local, version-controlled policy is a deliberate trust
decision — but it is a real escape hatch and is recorded in the audit decision reason.

`RunPolicyTest` (test.go) must call `ApplyPolicyOverlay` on top of the engine result so
`policy test` reflects real enforcement. `beekeeper check` (handler.go) must load
`~/.beekeeper/policies/*.json` via `policyloader.LoadPolicyFile`/a new loader helper,
then apply the overlay to its engine decision before emitting the final decision.
Loading failures are non-fatal for individual malformed files (skip with a logged
warning) but the directory load itself must fail closed only if it cannot be read at
all AND fail_mode is closed — match the existing handler fail-closed conventions.
</design>

- T-09-32 (Spoofing, V6): a misconfigured self-hosted PubKey must fail closed, never silently fall back to the embedded key (CR-01 fix). Mitigated by the decode-or-fail-closed requirement in 09-06-02.

</threat_model>
