<objective>
Close code-review blocker CR-01 (data loss in the live feed). `tailFrom` in watcher.go drains a `bufio.Scanner`, then computes the new offset from `f.Seek(0,1)` — the position of the last byte the reader BUFFERED, not the end of the last complete line. When an audit record is mid-write (no trailing newline yet — the common case for an append-in-progress NDJSON log), the scanner discards the partial line but the returned offset jumps past it, so on the next tick the offset starts beyond the now-completed record and that record is never emitted to the panel. A second bug: the `newOffset == 0` fallback resets the offset to the full file size, which can skip the entire remaining tail.

This is a correctness defect in the dashboard's primary live data feed — it directly undermines SC-1's "real-time" guarantee and starves the gap-1 activity feed (08-06) of events that arrive concurrently with the 1s ticker / fsnotify reads.

This plan rewrites `tailFrom` to advance the offset only past complete, newline-terminated lines (a `bufio.Reader` + `ReadString('\n')` loop that stops at the last newline and leaves a partial trailing line for the next tick), and removes the incorrect `newOffset == 0` size-jump fallback. It adds regression tests proving a partial trailing record is re-read and emitted once its newline lands.

Wave 1 — touches only watcher.go and a new watcher_test.go. Disjoint from 08-06 (alerts_panel.go), 08-08 (model/health/base), and 08-07 (policy/model). Independent of all other gap plans.
</objective>

|-----------|----------|-----------|-------------|-----------------|
| T-08-09-01 | Tampering | A malformed/partial NDJSON line stalls the feed or is silently lost | mitigate | Complete malformed lines are skipped but still advance the offset (no infinite re-read); partial lines are left pending and re-read once completed (no loss) — this IS the CR-01 fix |
| T-08-09-02 | Denial of Service | A line that never receives its newline holds the offset forever, blocking later records | accept | This is correct NDJSON tailing semantics: a never-terminated line means the writer crashed mid-record; the fallback ticker re-checks every 1s and emits as soon as the newline lands.
