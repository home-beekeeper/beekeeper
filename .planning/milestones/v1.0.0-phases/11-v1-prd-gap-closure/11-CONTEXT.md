<decisions>
## Locked fixes (grep-confirmed against current code)

### #1 — Gateway prompt-injection scan no-ops (SERIOUS, security)
`internal/gateway/proxy.go:288` and `:346` call `ScanProxiedResponse(ctx, "", respBytes, h.scanner)` with an EMPTY toolName; `llamafirewall.ShouldScanPrompt("")` returns false (only `web_search`/`read_file` match), so PromptGuard 2 never runs on the gateway. **Fix:** pass the real tool name (from the proxied JSON-RPC tool call) into `ScanProxiedResponse`. PromptGuard must actually run for eligible tools on the gateway path (LLMF-02).

### #2 — Layered config not used by enforcement (SERIOUS, CODE-05 SC2)
`newCheckCmd` (main.go:264), `newGatewayCmd` (1056), `newWatchCmd` (467), `newScanCmd` (540) all call single-file `config.Load`. `resolveConfig`/`LoadLayered` is only used by `diag` + the self-quarantine guard. **Fix:** route enforcement commands through the layered loader so system→user→project→`BEEKEEPER_*`→flags all apply; a project `.beekeeper/config.json` must override user config for a live `beekeeper check` decision WITHOUT env vars. Promote `resolveConfig` (currently in `cmd/beekeeper/diag.go`) to a shared cmd helper.

### #3 — Linux eBPF bindings are stubs (SERIOUS, locked CLAUDE.md constraint)
`internal/sentry/linux/bpf_beekeeper_{exec,net}_bpfel.go:40` return an error stub; no compiled `.o` bytecode is committed, so Linux Sentry degrades to fanotify-only (Tier2) and SLNX-02/04 eBPF ingestion never loads. **Constraint:** bytecode generation needs Linux + clang/llvm — it CANNOT be produced or verified on this Windows dev machine; it is a CI/Linux build step (the project already mandates an eBPF CI matrix). **Fix (achievable here):** wire `go generate ./internal/sentry/linux/...` into the Linux build pipeline — a `make generate` target, a CI step before `go build` on the Linux job, and a GoReleaser before-hook (Linux only) — so released Linux binaries embed real bytecode. Keep the loader fail-closed (clear error) when bytecode is absent. Document in CLAUDE.md / a build note that bytecode is CI-generated, never committed, never runtime-compiled. Do NOT attempt local bytecode generation.

### #4 — Catalog-delta-triggered scan not wired (MODERATE, CTLG-06)
`catalogs watch` `onDelta` callback (main.go:369–381) only logs. **Fix:** on delta, run `catalog.Sync` (refresh) then `scan.Scan` so a new `threat_intel/` entry triggers an immediate machine sweep, per PRD §5.4 / CTLG-06.

### #5 — `catalogs diff` command missing (MODERATE, PRD §10)
PRD §10 lists `beekeeper catalogs diff`; not registered in `newCatalogsCmd`. **Fix:** add `catalogs diff` showing the delta between the cached catalog state and the latest fetched/last-synced state (reuse `catalog` state/sanity delta logic).

### #6 — Catalog signature verification is presence-only (MODERATE, CTLG-07/09)
`catalog/verify.go` checks `CatalogSignature != ""` (not real crypto) for Bumblebee/OSV/Socket; `SourcesDissented` is always empty. **Fix:** implement real Ed25519 verification when a catalog entry/source carries a signature AND a trusted key is configured for that source; when no signature/key (e.g. Bumblebee unsigned), keep the unsigned→warning-only-regardless-of-corroboration behavior (PRD §12.3) — do not hard-fail unsigned sources. Populate `SourcesDissented`/`Dissented` in the corroboration provenance so forensics can trace dissent.

## Constraints (unchanged)
- `internal/policy` stays pure. Fail-closed default. Single static binary, no CGO in core. Build-tag pairs paired.
- `go build ./...`, `GOOS=windows go build ./...`, `go test ./...` must stay green.
- eBPF: never compile at runtime; CI-generated bytecode only (this phase cannot generate/verify it locally).
