<objective>
Build the OSV catalog source as an HTTP adapter over the public OSV REST API (`POST https://api.osv.dev/v1/query`), with a 24h-TTL disk cache, producing `policy.CatalogMatch` results for corroboration. This is the SECOND independent signed source that PLCY-01 corroboration requires to escalate a warn to a block.

Purpose: Phase 1 had Bumblebee only — so corroboration could never reach 2 signed sources. OSV is the public, no-auth source that makes two-source enforcement real. The adapter resolves all I/O OUTSIDE `internal/policy`, handing the pure engine a `[]policy.CatalogMatch` (PATTERNS "I/O adapters return pre-resolved inputs" rule).

Output: `internal/catalog/osv.go` + test. Depends on Plan 01 for `policy.CatalogMatch` provenance fields and the `MultiCatalogLookup` contract shape. The aggregator that combines Bumblebee+OSV+Socket into one `MultiCatalogLookup` is Plan 08 (Wave 3); this plan delivers the OSV piece in isolation.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OSV API → adapter | Remote JSON response is untrusted input (bounded read, schema-tolerant parse) |
| adapter → disk cache | Cache file under ~/.beekeeper/catalogs/osv/ could be tampered to fabricate or suppress matches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Tampering | OSV response parsing | mitigate | 4MB LimitReader bound; tolerant JSON parse; non-200 → error (no fabricated allow) |
| T-02-04-02 | Spoofing | ecosystem name mismatch silently misses vulns | mitigate | Explicit case-sensitive osvEcosystem table; unmapped ecosystem returns (nil,nil) rather than querying with a wrong name that would 404/empty |
| T-02-04-03 | Denial of Service | OSV API unavailability stalls check hot path | mitigate | Caller-supplied ctx carries the 5s hook deadline; cache-first avoids network on the common path; failure degrades to no-match for this source only (corroboration still works with remaining sources) |
| T-02-04-04 | Tampering | poisoned disk cache | accept | Cache dir is owner-only (0o700); a local attacker with write access already has broader compromise; documented as accepted in threat model |
</threat_model>
