<objective>
Replace the `realMetadataFetch` stub (internal/gateway/drift.go:57-63, returns an empty map) with a real registry metadata query so the gateway weekly drift check emits live `record_type:"version_drift"` records. All the wiring (scheduler, IsMajorDrift, emitVersionDrift, async/one-in-flight bounding) is already built and tested via the injected `metadataFetchFn` seam; only the production fetch is a stub.

Purpose: Without a real fetch, production drift never emits a record — the operator never learns a new pnpm/bun major shipped. This is the last incomplete production path in the v1.2.0 nudge feature.

Constraints (hard): drift is INFORMATIONAL ONLY. The fetch lives in the gateway adapter (never in a pure package). It is FAIL-OPEN — any fetch/HTTP/parse error emits no record and never blocks or panics the gateway (it runs off the request hot path in a bounded goroutine). Floors are NEVER auto-bumped (auto-update stays Out-of-Scope, PRD §7.1). No new heavy dependency — stdlib `net/http` only, reusing the existing registry-fetch idiom.

Output: `realMetadataFetch` queries the npm registry for the latest published pnpm (and bun) version behind an overridable base URL; httptest-backed tests prove the parse, the drift emit, and fail-open on error.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| gateway daemon → npm registry (registry.npmjs.org) | DRIFT-01 introduces a NEW outbound network call from the long-lived gateway to an external registry. The registry response is untrusted input (could be large, malformed, slow). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-10 | Denial of Service (slow/large response) | realMetadataFetch HTTP read | mitigate | A slow or huge registry response could hang or balloon the drift goroutine. Fix: *http.Client with an explicit short timeout (~5s), io.LimitReader cap on the body (dist-tags is tiny — ~256KB cap), and the scheduler's existing 30s per-check ctx + one-in-flight atomic bound (WR-04). Drift runs off the request hot path. |
| T-09-11 | Information Disclosure / SSRF | registry base URL | mitigate | The base URL is a hardcoded constant (registry.npmjs.org), overridable ONLY by the test package var — never derived from agent input or untrusted config, so there is no request-forgery surface. The request is a fixed GET to a fixed host path; no user-controlled URL components beyond the static PM names {"pnpm","bun"}. |
| T-09-12 | Tampering (poisoned drift → floor bump) | checkDrift / emitVersionDrift | mitigate | A malicious registry response claiming a huge "latest" must NOT cause a floor change. Fix: drift is informational-only — emitVersionDrift writes an audit record and NEVER mutates VersionFloors (PRD §7.1). Tests assert floors are unchanged after checkDrift. |
| T-09-13 | Denial of Service (fetch failure cascades) | realMetadataFetch error path | mitigate (fail-open by design) | A registry outage must not block or crash the gateway. Fix: per-PM fail-open omission + checkDrift's existing fail-open on error; the drift check is best-effort and advisory. Fail-open is correct HERE specifically because drift is informational and off the enforcement path — it never gates a tool call. |

The fail-open disposition is intentional and scoped: it applies ONLY to the informational drift check, off the enforcement hot path. It does NOT touch the fail-CLOSED contract of `beekeeper check` / catalog / SPATH enforcement. No tool call is ever allowed or blocked as a result of this network call.
</threat_model>
