<objective>
Make the critical-severity override operator-configurable (CORR-01 Q3) by adding an optional `critical_block_at` field to the existing `corroboration_threshold` policy-file rule type. The field merges into `SeverityOverrides["critical"]` via the established non-zero-override pattern, is bound-checked (`>= 1`) at load time, and requires no `schema_version` bump.

Purpose: Operators must be able to revert to 2-source behavior (`critical_block_at: 2`) without a Beekeeper release if false positives emerge (PITFALLS.md severity-inflation recovery path). Hardcoding-only would force a release to tune.
Output: Extended `PolicyRule` (loader.go), load-time bound check (validate.go), threshold merge in `ThresholdsFromPolicyFiles` + `thresholdsFromPolicyFile` (test.go), and a new loader unit test.
</objective>
