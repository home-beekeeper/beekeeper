|---------|-------------|----------|---------------------|
| "Green ≠ proven" | All "23 packages green" reported this project is **Windows-local `go test` WITHOUT `-race`** — no eBPF, no kernel VMs, no live eslogger, no fuzz gates, no integration VMs. They are CI-only and CI never ran. | blocking | Do NOT call v1.0.0 "tested" until the first **GitHub Actions** run is green. | "Wiring exists ≠ effective" | Two serious gaps shipped because verification confirmed a symbol/call *existed* but not that it was *effective*: gateway `ScanProxiedResponse` was called with an empty tool-name arg (no-op); `LoadLayered` existed but enforcement commands never called it. | advisory | When verifying integration, grep for the **effective call arguments and actual callers**, not just symbol presence. |
| eBPF can't build locally | Linux eBPF bytecode needs clang/llvm/libbpf on Linux; the dev box is Windows. Bytecode is **CI-generated** (`go generate` in the Linux CI job + GoReleaser before-hook), never committed. | advisory | Never attempt local eBPF bytecode generation. The loader stub fails closed if bytecode is absent; that's expected off-CI. |
