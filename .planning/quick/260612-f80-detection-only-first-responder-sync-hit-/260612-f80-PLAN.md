<reconciliation_notes>
## Corroboration tier vs >=2 auto-quarantine (READ FIRST)

The corroboration engine (`internal/policy/corroboration.go`) today reserves the quarantine
*recommendation* (`quarantine=true`) for the QuarantineAt tier (3 signed sources): 1 -> warn,
2 -> block, 3 -> block+quarantine. The maintainer wants AUTO reversible-quarantine at **>= 2**.

DO NOT change the policy engine's quarantine semantics. Instead:
- `policy.Evaluate` / `corroborate` stay exactly as they are (pure, untouched).
- The NEW first-responder layer reads `Decision.CorroborationCount` (the distinct SIGNED-source
  count) and fires AUTO reversible-quarantine when `CorroborationCount >= AutoQuarantineThreshold`.
- `AutoQuarantineThreshold` is a NEW config knob defaulting to **2** (the "block" tier). This is safe
  because the auto action is REVERSIBLE (a dir move) and the DESTRUCTIVE purge stays human-gated.

## Anti-pattern from the handoff: default-vs-clamp-floor coupling

Mirror `CatalogSyncConfig` exactly (`internal/config/config.go` ~265-364), but note the bug class it
illustrates: a clamp FLOOR must not silently rewrite a legitimate default. For AutoQuarantine the
threshold is an integer corroboration count: clamp to `[1, 3]` (1 = warn-tier opt-in for the
paranoid; 3 = the engine's own quarantine tier; default 2). A zero/absent value resolves to the
default 2, NOT to the clamp floor 1 — keep the "absent -> default" and "present-but-out-of-range ->
clamped" paths distinct, exactly like `parseClampCatalogSyncInterval` vs `DefaultCatalogSyncConfig`.

## internal/policy purity

No new code in `internal/policy`. The threshold comparison happens in `internal/watch` /
`internal/scan`, which already import policy. policy remains I/O-free.

## Honesty invariants (bake into EVERY task)
- Scan inventory (Pollen/Bumblebee) stays READ-ONLY. Cross-reference reads the index + reads
  installed package dirs; it never removes/disables/edits a package.
- Quarantine is a reversible MOVE (os.Rename) + manifest; the only destructive op is Purge, which is
  human-gated (CLI prompt / TUI [P]).
- Sentry stays DETECTION-ONLY: the target list tightens thresholds and escalates labels; it NEVER
  kills, isolates, or cuts network.
- Fail-closed: any error in the first-responder path logs + leaves the artifact in place (no
  half-move), and never downgrades a block to allow.
- Auto-quarantine is opt-in (config) with a dry-run-default option; every action is audited.

## Windows / CRLF gotchas
- Any new golden/byte-exact fixture needs a `.gitattributes` `text eol=lf` line (mirror the existing
  `internal/check/testdata/deny/*.golden` entry).
- B's Sentry rule tests are cross-OS-pure (live in `internal/sentry`, no build tag) so they run on
  Windows directly. If any test file lands under `internal/sentry/linux|darwin` (it should NOT for
  this slice), compile-validate it with `GOOS=linux go vet ./...` / `GOOS=darwin go vet ./...`
  because `go build` does NOT compile test files.
