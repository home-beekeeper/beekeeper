<objective>
Close FRB-05 by wiring the local catalog overlay into the live `beekeeper check`
hot path, then lock the wiring with a RunCheck-level regression test.

A v1.4.0 milestone audit confirmed a BLOCKER: the overlay WRITE half works
(`catalogs_daemon.go:~169` calls `catalog.AddLocalOverlayEntry`, producing

the upstream feed") does not happen. Every existing overlay test calls
`NewMultiIndexWithOverlay` DIRECTLY, never through a live `RunCheck`, which is why
the green suite missed it. A test that asserts the component is not a test of the path.

Purpose: make the confirmed-malicious local overlay actually enforce at check time
(FRB-05), with a regression test that drives the real handler so the wiring cannot
silently regress again.
Output: overlay-aware `runCheck`; one RunCheck-level overlay-escalation test; one
no-overlay-file regression guard.

CORRECTNESS NOTE (read before coding the test assertion): overlay entries are
UNSIGNED (`CatalogSignature == ""` â€” see `internal/catalog/local_overlay.go` doc and
`buildOverlayEntry` in `catalogs_daemon.go`, which leaves signature empty per CTLG-07
warn-only). `internal/policy/corroboration.go` escalates to `block` only on
`signedCount >= effectiveBlockAt && hasSignedSource` (multi.go LookupAll forces the
overlay match's `Signed` to remain false). Therefore an overlay match alone CANNOT
produce `block` â€” it produces `warn`. The scope guidance's "1-source warn â†’ block"
framing does not hold under the real corroboration semantics. The honest,
load-bearing escalation the overlay produces is **allow â†’ warn**: a package with NO
catalog match (decision `allow`) becomes `warn` once the overlay contributes a
`local-overlay` source. That allowâ†’warn escalation fully proves the wiring and does
NOT fabricate a signed overlay (which would contradict the documented CTLG-07
invariant). The test asserts exactly that, plus the no-overlay regression guard.
