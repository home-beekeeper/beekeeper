<objective>
Fix the fail-closed WIRING gap: every agent-harness hook installer currently writes the BARE command name `beekeeper` (e.g. `beekeeper check --hook claude-code`). When `beekeeper` is not on the PATH snapshot the harness captured at launch (installed/upgraded while the agent is running, or any PATH drift), the harness shell resolves the hook to nothing and returns exit 127 ("command not found"). Exit 127 is NON-BLOCKING (only exit 2 blocks), so the agent silently runs UNPROTECTED until restart. There is no beekeeper process to "fail closed" because the shell can't find it. This was hit live on the maintainer's machine.

The fix embeds the running binary's absolute path (resolved via `os.Executable()` at install time, `filepath.ToSlash`'d and quoted) into EVERY harness's hook command, with a sane fallback to the bare name if resolution fails. Because the command now varies by machine, exact-full-string detection breaks — so detection/idempotency/uninstall move to a STABLE INVARIANT SUFFIX match (`check --hook <harness>` / `audit-record`), and re-install MIGRATES an old bare-name entry to the new abspath form in place.

Purpose: Restore beekeeper's "fail closed by default" guarantee at the wiring layer — the hook resolves and runs regardless of the harness PATH snapshot.
Output: A shared command-builder + suffix-matcher helper (`internal/hooks/command.go`), all 13 harness installers routed through it, and a green test suite proving abspath generation, fallback, idempotency, migration, uninstall, and foreign-hook preservation.

SCOPE BOUNDARY (locked): beekeeper Go source ONLY. Do NOT touch install.ps1 / the web installer (separate follow-up in beekeeper-web). Do NOT build or install the binary. The decision to embed the absolute path (machine-specific settings.json) is LOCKED by the maintainer — do not re-litigate or add a portability/bare-name config flag.
</objective>
