# Phase 12 — UI Design Contract

> Visual and interaction contract for the Beekeeper v1.3.0 web site design system.
> Generated by gsd-ui-researcher. Every web phase (13–18) inherits this contract.
> Design decisions are LOCKED from beekeeper-docs.html (authoritative mockup).

## Scope Fence

This contract covers the **design system only**: color tokens, typography, spacing, shadcn/ui
configuration, the theme provider, the reduced-motion provider, focus/keyboard treatment,
and the base shadcn component set to install.

The following are **explicitly out of scope** for Phase 12 and must NOT be designed here:
- Marketing home page sections (Phase 15)
- Docs page layout and sidebar (Phase 13)
- Changelog route layout (Phase 14)
- 3D/R3F scene design (Phase 16)
- SEO metadata (Phase 17)
- MDX prose content (Phase 18)

### Why this order

| Position | Entry | Why here |
|----------|-------|----------|
| 1st | `@import "tailwindcss"` | Tailwind v4 engine must initialize before any other CSS so its utility classes are available to all subsequent rules |
| 2nd | `@import "fumadocs-ui/css/shadcn.css"` | Fumadocs' shadcn bridge sets its own CSS variable defaults; must come before our overrides or our tokens get clobbered |
| 3rd | `@import "fumadocs-ui/css/preset.css"` | Fumadocs component styles (sidebar, TOC, breadcrumbs) may reference the variables set in shadcn.css; must follow it |
| 4th | `@source` | Directs Tailwind's content scanner at Fumadocs' compiled JS for class purging; must be after imports that define what to scan |
| 5th | `@theme inline { … }` | Beekeeper's own tokens. `inline` means Tailwind exposes them as utility classes AND CSS variables. Placed last so Beekeeper values WIN over both shadcn and Fumadocs defaults |
| 6th | `:root` / `.light` | Raw CSS custom properties (not Tailwind utilities) for non-Tailwind contexts. Placed after `@theme` so Tailwind utilities take precedence in utility-class contexts |

## Font Families

### Loading strategy: `next/font/google`

Self-hosted at build time via `next/font/google`. This works under `output: 'export'`
because Next.js downloads the font files during build and emits them into `out/_next/static/`.
No external font request at runtime.

```tsx
// web/app/layout.tsx (Phase 12 replaces the Geist implementation)
import { Inter, JetBrains_Mono } from "next/font/google";

const inter = Inter({
  subsets: ["latin"],
  variable: "--font-inter",
  display: "swap",
  // Include the optical size axis for display use at large sizes
  axes: ["opsz"],
});

const jetbrainsMono = JetBrains_Mono({
  subsets: ["latin"],
  variable: "--font-jetbrains-mono",
  display: "swap",
});
```

### "Inter Display" substitution

`Inter Display` is NOT a separate Google Fonts family. The Inter variable font on Google Fonts
includes the optical size axis (`opsz`). At large sizes (≥ 32px), load Inter with `opsz` at its
maximum value (32) to approximate the tighter, taller proportions of Inter Display.

**Implementation:** Use CSS `font-optical-sizing: auto` on heading elements. The browser uses
`opsz` automatically when the axis is present. Combined with the tight `letter-spacing` values
in §Typography, this matches the mockup's display aesthetic.

```css
h1, h2, h3, .display {
  font-family: var(--font-display); /* resolves to Inter via @theme */
  font-optical-sizing: auto;
}
```

### globals.css Geist removal

Phase 11 left `globals.css` referencing `--font-geist-sans` and `--font-geist-mono` in its
`@theme inline` block. Phase 12 owns globals.css and replaces that entire file with the
canonical order above. The Geist `className` props on `<html>` in layout.tsx are also removed.

## Semantic Color Mappings

**These mappings are binding across all web phases (13–18).** Do not use a semantic color
outside its defined role without updating this contract.

| Color | Token | Semantic Role | Reserved For |
|-------|-------|---------------|-------------|
| Amber | `--amber` | **Brand accent** | Logo mark, section eyebrows, active nav underline, quickstep numbers, star-ratings, primary CTA background (dark theme), install tab active indicator, glyph icons |
| Teal | `--teal` | **Interactive / links** | All anchor tags, hover states on interactive elements, CLI prompt `$` character, keyboard shortcut labels in keybars, focus ring (via `--color-ring`), interactive pill-mini badges |
| Red | `--red` | **Threat** | Critical severity badges, threat row highlighting, incident card borders/backgrounds, sentry-firing indicators, pulse animations on critical events, changelog breaking-change callout |
| Coral | `--coral` | **Response / destructive hints** | High-severity badges, response layer labels, proactive defense callouts, destructive action key hints (never for confirm buttons — use explicit destructive variant) |
| Green | `--green` | **Healthy / success** | Status pips (healthy), "live synced" indicators, success states, allowed-action confirmations, checkmarks in self-defense callout, green-pulse ring animation |
| Purple | `--purple` | **Secondary accent** | Reserved — currently used only for inline `code` elements in dark theme. Do not introduce new uses in v1.3.0 |

### The 60/30/10 split

In dark theme:
- **60% dominant:** `--dim` (`#8b95a5`) is the most-used text color (body copy, descriptions)
- **30% secondary:** `--fg` (`#d4dae3`) for emphasis within body text, nav labels, feature names
- **10% accent:** `--amber`, `--teal`, `--red`, `--coral`, `--green` each used sparingly

In light theme: same proportional intent with the derived values.

## Animations and Motion

### Motion tokens

| Animation | Value | Usage |
|-----------|-------|-------|
| Color/border transition | `0.15s ease` | Nav links, borders, buttons — all hover states |
| FAQ icon rotate | `0.2s ease` | `<details>` expand/collapse |
| Pulse (threat/critical) | `1.1–1.2s infinite` opacity 1→0.35 | Critical status pip, demo titlebar live dot |
| Green pulse (live-synced) | `2s infinite` box-shadow ring | Threats table "live synced" indicator |
| Blur transition (demo overlay) | `0.25s` opacity + filter | Demo screen blur on overlay open |

### Reduced-motion gate (DSYS-03)

All animations must be fully gated by `prefers-reduced-motion: reduce`. This is implemented
via a `ReducedMotionProvider` React context (see §ReducedMotionProvider Contract below).

**CSS-side gate (belt and suspenders — applies even before JS hydrates):**

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

This global rule goes inside `:root` section of globals.css, after the theme tokens.

## Theme Toggle: No-FOUC Contract (DSYS-02)

### Strategy: `next-themes` with class strategy

```bash
# Install from web/
pnpm add next-themes
```

**Configuration:**

```tsx
// web/app/providers.tsx  (new file — Phase 12 creates this)
"use client";
import { ThemeProvider } from "next-themes";
export function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider
      attribute="class"           // adds .dark or .light to <html>
      defaultTheme="system"       // respects prefers-color-scheme on first visit
      enableSystem={true}
      storageKey="bk-theme"       // localStorage key
      disableTransitionOnChange   // prevents flash during theme switch
    >
      {children}
    </ThemeProvider>
  );
}
```

```tsx
// web/app/layout.tsx  (Phase 12 replacement)
import { Inter, JetBrains_Mono } from "next/font/google";
import { Providers } from "./providers";
import "./globals.css";

const inter = Inter({ subsets: ["latin"], variable: "--font-inter", display: "swap", axes: ["opsz"] });
const jetbrainsMono = JetBrains_Mono({ subsets: ["latin"], variable: "--font-jetbrains-mono", display: "swap" });

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html
      lang="en"
      className={`${inter.variable} ${jetbrainsMono.variable}`}
      suppressHydrationWarning   // REQUIRED: next-themes writes class before hydration
    >
      <body className="min-h-full flex flex-col antialiased">
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}
```

### No-FOUC mechanism

`suppressHydrationWarning` on `<html>` combined with next-themes' inline script (injected
automatically at the bottom of `<head>`) reads `localStorage["bk-theme"]` or
`prefers-color-scheme` and adds the `.dark` or `.light` class BEFORE the first paint.
This prevents any flash of the wrong theme.

**Critical:** `suppressHydrationWarning` is REQUIRED on the `<html>` element. Without it,
React will throw a hydration mismatch because the server renders without the `.dark`/`.light`
class but the client adds it via the inline script.

### CSS class wiring

The `.dark` class is applied to `<html>`. Dark theme = no `.light` class (dark is the
CSS default in `:root`). Light theme = `.light` class added.

```css
/* In globals.css — theme class wiring */
/* Dark: defaults in :root (dark is canonical brand) */
/* Light: .light class overrides */
html:not(.dark) .light-only { display: block; }
html.dark .dark-only { display: block; }
```

**Fumadocs integration:** Fumadocs `RootProvider` also needs theme awareness. Wire it inside
`Providers`:

```tsx
import { RootProvider } from "fumadocs-ui/provider";
// RootProvider wraps ThemeProvider OR sits inside it.
// Phase 13 determines exact nesting — Phase 12 establishes the ThemeProvider as the
// outermost context so RootProvider can consume it.
```

## ReducedMotionProvider Contract (DSYS-03)

### Scope

This provider gates BOTH:
1. CSS animation (via `data-reduced-motion` attribute on `<html>`)
2. Future Phase 16 R3F/Three.js animation (via React context `useReducedMotion()` hook)

It is the single source of truth for motion preference site-wide.

### Contract

```tsx
// web/lib/reduced-motion.tsx  (Phase 12 creates this)
"use client";
import { createContext, useContext, useEffect, useState } from "react";

interface ReducedMotionContextValue {
  /** True when the user prefers reduced motion OR has explicitly disabled motion */
  prefersReducedMotion: boolean;
}

const ReducedMotionContext = createContext<ReducedMotionContextValue>({
  prefersReducedMotion: false,
});

export function ReducedMotionProvider({ children }: { children: React.ReactNode }) {
  const [prefersReducedMotion, setPrefersReducedMotion] = useState(false);

  useEffect(() => {
    const mq = window.matchMedia("(prefers-reduced-motion: reduce)");
    setPrefersReducedMotion(mq.matches);
    // Sync the data attribute for CSS selectors
    document.documentElement.dataset.reducedMotion = mq.matches ? "true" : "false";

    const handler = (e: MediaQueryListEvent) => {
      setPrefersReducedMotion(e.matches);
      document.documentElement.dataset.reducedMotion = e.matches ? "true" : "false";
    };
    mq.addEventListener("change", handler);
    return () => mq.removeEventListener("change", handler);
  }, []);

  return (
    <ReducedMotionContext.Provider value={{ prefersReducedMotion }}>
      {children}
    </ReducedMotionContext.Provider>
  );
}

/** Hook consumed by all animation components and the Phase 16 R3F canvas */
export function useReducedMotion(): boolean {
  return useContext(ReducedMotionContext).prefersReducedMotion;
}
```

### Phase 16 consumption contract

Phase 16's `Hero3D.tsx` MUST check `useReducedMotion()` before mounting the R3F canvas:

```tsx
// Phase 16 — Hero3D.tsx (contract defined here, implemented in Phase 16)
import { useReducedMotion } from "@/lib/reduced-motion";
import dynamic from "next/dynamic";

const HeroCanvasInner = dynamic(() => import("./HeroCanvasInner"), { ssr: false });

export function Hero3D() {
  const reduced = useReducedMotion();
  if (reduced) return <HeroSVGFallback />; // Static SVG; never mounts WebGL
  return <HeroCanvasInner />;
}
```

**Rule:** The canvas MUST NOT be mounted and then paused. Under reduced motion, `Hero3D`
renders the static SVG fallback and NO canvas element appears in the DOM. This satisfies
WCAG 2.3.3 (Animation from Interactions).

### Nesting in Providers

```tsx
// web/app/providers.tsx
export function Providers({ children }) {
  return (
    <ThemeProvider attribute="class" defaultTheme="system" enableSystem storageKey="bk-theme" disableTransitionOnChange>
      <ReducedMotionProvider>
        {children}
      </ReducedMotionProvider>
    </ThemeProvider>
  );
}
```

## Focus / Keyboard / WCAG-AA Contract (DSYS-04)

### Focus ring

**All interactive elements** must have a visible focus indicator using the teal ring token.
shadcn/ui components use the `--color-ring` CSS variable for `outline`. Phase 12 sets this
to `--teal` in both themes.

```css
/* In @theme inline block */
--color-ring: var(--color-bk-teal); /* dark: #39c5cf / light: #0e7e8a */
```

**Tailwind utility for custom focusable elements:**

```
focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-bk-teal
```

**Do not suppress focus rings** with `outline: none` unless a custom focus indicator replaces
the native one with equal or greater visibility.

### Keyboard navigation rules

| Element | Keyboard requirement |
|---------|---------------------|
| Nav links | Tab-reachable; Enter activates |
| Theme toggle button | Tab-reachable; Enter/Space toggles |
| Install command tabs | Tab-reachable; Arrow keys switch tabs (ARIA tablist pattern) |
| FAQ `<details>` | Tab-reachable; Enter opens/closes |
| Code copy buttons | Tab-reachable; Enter copies |
| All CTA buttons | Tab-reachable; Enter activates |
| Skip link | First tab stop; links to `#main-content` |

**Skip link** is required (WCAG 2.4.1). Phase 12 adds it to layout.tsx:

## Fumadocs CSS Integration Notes

### What Fumadocs ships

`fumadocs-ui` ships two CSS files that must both be imported:

- `fumadocs-ui/css/shadcn.css` — remaps Fumadocs' internal component tokens onto shadcn
  CSS variable names (`--primary`, `--background`, etc.)
- `fumadocs-ui/css/preset.css` — component-level styles (sidebar width, TOC layout,
  search dialog, breadcrumbs, code block)

### Why Beekeeper tokens win

Because our `@theme inline` block comes AFTER both Fumadocs CSS imports, our values for
`--color-background`, `--color-primary`, `--color-ring`, etc. override whatever Fumadocs
set. The `@source` directive ensures Fumadocs' Tailwind classes survive tree-shaking.

### Fumadocs versions (Phase 12 installs)

Do NOT install Fumadocs in Phase 12. Phase 13 installs:
- `fumadocs-core@^16.7.0`
- `fumadocs-ui@^16.x`
- `fumadocs-mdx@^15.0.11` (NOT 14.x — see 11-RESEARCH.md correction)

Phase 12 writes the import stubs in globals.css as comments so Phase 13 can uncomment
them after install, or writes them actively if doing a combined init in one phase.

**Recommended approach for Phase 12:** Write the full canonical globals.css including the
Fumadocs import lines. When `pnpm build` runs before Phase 13 installs Fumadocs, these
imports will fail. Phase 12 must therefore either:

1. (Preferred) Keep the Fumadocs imports as comments and instruct Phase 13 to uncomment, OR
2. Install a minimal Fumadocs stub now (just `fumadocs-ui` for the CSS files)

**Decision (made here):** Use option 1. The globals.css written in Phase 12 will have the
Fumadocs imports as comments. Phase 13 uncomments them after installing the packages.
The `@source` directive is also commented until Phase 13 activates it. The `@theme inline`
block is fully active in Phase 12 (it does not reference Fumadocs packages).
