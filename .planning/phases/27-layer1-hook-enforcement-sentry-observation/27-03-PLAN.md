# Plan 27-03 — Make the shim real (maintainer decision at Gate 1)

## Why

Gate-1 review found the shim enforces NOTHING today: its wrapper runs
`beekeeper check --tool "npm" --args "$@"`, which the `--tool` path turns into a
tool call of shape `{tool_name:"execute", tool_input:{command:"npm", args:[...]}}`.
The catalog engine's `extract` and the posture adapter both read ONLY
`tool_input["command"]` (`"npm"`) and ignore `args`, so no install is identified
and `beekeeper check` returns allow. The shim is a routing shell that no-ops
enforcement. Maintainer decision: make it real (wire it + honest docs), which
also removes the false-protection foot-gun.

## Tasks

### Task 1 — reconstruct the command on the --tool path (the wiring fix)
In `cmd/beekeeper/main.go`, the `--tool` branch of `newCheckCmd`:
- Extract a testable helper `buildShimToolCall(tool string, args []string) map[string]any`.
- It reconstructs the full shell command (`tool + " " + strings.Join(args, " ")`)
  and shapes the tool call as a BASH call:
  `{tool_name:"Bash", agent_name:"shim", tool_input:{command:"npm install left-pad"}}`.
  Bash is the correct model: the shim intercepts a shell invocation of a package
  manager, and Bash is the shape both `extract` (catalog) and `evaluatePosture`
  parse. Still injection-safe: the command is a JSON string value built by
  json.Marshal and parsed by the pure pkgparse (no shell execution).
- Preserve the TM-A-04 arg-merge (flag --args first, then cobra positional
  overflow) and the CR-03/CR-04 injection-safety property.

### Task 2 — honest boundary copy (the shim now enforces, with caveats)
`internal/posture/boundary.go`: update the shim sentence in BoundaryStatement and
BoundaryShort. The shim is no longer "roadmap that does nothing" — it now enforces
the same posture for human-run installs, BUT it is experimental and limited:
- it only covers package managers invoked through the shimmed PATH;
- it is bypassable by calling a tool by its absolute path;
- it requires adding the shim directory to PATH.
Keep "experimental" framing; drop any wording that implies it does not work at
all OR that it is a complete machine-wide guarantee. boundary_test.go must still
pass (it checks the statement names hook/sentry/gateway/shim and frames the shim
as experimental/roadmap).

### Task 3 — honest shim help/docs
`internal/shim/shim.go` package doc + the `beekeeper shim` command help
(`cmd/beekeeper/main.go newShimCmd`): state plainly that the shim runs
`beekeeper check` before each intercepted install (so catalog + install posture
apply), that it is experimental, and that it is bypassable by absolute-path
invocation and requires the PATH prepend. No wording that implies a complete
guarantee.

### Task 4 — tests (prove the path)
- Unit test `buildShimToolCall` (cmd/beekeeper): `("npm", ["install","left-pad"])`
  → `tool_name:"Bash"`, `command:"npm install left-pad"`; no-arg → `command:"npm"`.
- Live-path test: drive the shim-produced tool call through `check.RunCheck`
  (either a cmd-level test that builds the shape via buildShimToolCall, or an
  e2e-tag test invoking the built binary `beekeeper check --tool npm --args install
  --args <fresh-pkg>`) and assert: a fresh (<24h) package WARNS (exit 0); a
  catalog-blocked package BLOCKS (exit non-zero). This proves the shim path now
  enforces, closing the latent gap.
