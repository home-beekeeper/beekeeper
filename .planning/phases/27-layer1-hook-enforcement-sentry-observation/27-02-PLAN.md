# Plan 27-02 — Sentry observation + the canonical boundary statement

## Goal

(1) Make Sentry observe and audit installs it sees (agent-run or human-run) as
DETECTION, never prevention (IPST-06). (2) Establish a single canonical
enforcement-boundary statement in code that every later surface (CLI help, the
`beekeeper posture` view, docs, web copy) references, so the honesty standard is
enforced from one source of truth (IPBND-01). This plan is the second half of
Gate 1: after it lands, the enforcement map + the exact boundary copy are ready
for the maintainer's review.

## Task 1 — canonical boundary statement (the source of truth)
- Add an exported constant in `internal/posture` (e.g. `BoundaryStatement`) with
  EXACTLY this text (no em dashes, sentence case; this is the PRD's boundary
  statement, lightly adapted):

  > Install posture is enforced pre-execution at the agent hook for hooked
  > harnesses that support it, inheriting each harness tier's caveats. For
  > harnesses with no pre-exec hook and for installs a person runs directly in a
  > terminal, it is observed and audited by the Sentry layer rather than
  > prevented. The MCP gateway is not a general install surface. The
  > package-manager shim that would extend pre-exec enforcement to every install,
  > including ones a user types directly, exists today as an experimental,
  > roadmap-tier surface and is not a headline v1.0 guarantee.

- Add a one-line short form too (for help text / compact UI):
  `BoundaryShort = "Posture is prevented at the hook for hooked harnesses only; everything else Sentry observes and audits, it does not prevent."`
- This constant is referenced (not re-typed) by the posture adapter comment
  (27-01), the `beekeeper posture` view (Phase 28), `beekeeper check`/posture help
  text, and the docs/web copy (Phase 30). Re-typed copies are a honesty-drift risk;
  keep ONE source.

## Task 2 — Sentry observes installs (detection only)
- Read `internal/sentry/` (the shared correlation engine + the per-OS event
  ingestion) to find where process-spawn events are handled and where Sentry writes
  its detection audit records. Sentry is DETECTION-ONLY by design (it must not
  block or kill).
- Recognize install-class processes (the same managers `pkgparse` knows: npm, pnpm,
  bun, yarn, pip/pip3, cargo, gem, composer, go, with an install/add/get/require
  verb) in the process-event path. When one is observed, emit a Sentry detection
  audit record clearly labeled as an OBSERVED install (detection, not prevention) —
  reuse the existing Sentry audit record machinery and the existing
  editor-/agent-descendant ancestry gating; do not invent a new privileged code
  path. The record notes the install (process attribution: pid/ppid/argv) and is
  labeled detection-only.
- HONEST SCOPE: Sentry records THAT an install happened (process attribution). It
  does NOT fetch the registry to compute release-age/lifecycle match in the
  privileged daemon (we do not add network I/O to the privileged tier). Posture
  matching is the hook's job. State this limitation in a code comment and carry it
  into the boundary docs — do not imply Sentry evaluates posture.
- If `internal/sentry` already audits every watched process spawn generically, then
  IPST-06 is mostly: ensure install-class spawns are covered + carry the
  detection-only label + reference the boundary statement. Keep the change minimal
  and honest; do not expand Sentry's scope beyond observation.

## Task 3 — tests
- A Sentry-rule/fixture test (mirror the existing synthetic process-tree fixtures)
  feeding an install-class process event and asserting a detection-only audit
  record is produced with the install attribution and the detection label, and that
  NO block/kill/quarantine action is taken (Sentry stays detection-only).
- A test asserting `posture.BoundaryStatement` is non-empty, em-dash-free, and names
  the four surfaces (hook, Sentry, gateway, shim-roadmap) so the honesty content
  cannot silently drift.

## Gate 1 (STOP after this plan)
After 27-01 + 27-02 are green, STOP and present to the maintainer:
1. The enforcement map AS IMPLEMENTED (hook = prevent for hooked Tier-1 incl. tier
   caveats; Sentry = observe/audit incl. human installs, detection-only; gateway =
   not a general install surface; shim = experimental/roadmap).
2. The exact `posture.BoundaryStatement` copy that will appear in product + docs.
3. The PRD divergences: (a) posture default action is WARN and FAIL-SOFT (warns on
   missing-timestamp/registry-error/timeout) at the hook, vs the pure evaluators'
   fail-closed block which remains for the extension scan/watch path; (b) the shim
   already existed and is kept+repointed+documented-as-roadmap rather than built or
   deleted.
Do NOT finalize docs/home/changelog (Phase 30) or the view/override (Phases 28-29)
until the maintainer ratifies the boundaries.
