## Warnings

### WR-01: `next start` script is inoperative with `output: "export"`

**File:** `web/package.json:9`
**Issue:** The `"start": "next start"` script requires a Node.js server to serve `.next/` build
output. With `output: "export"`, Next.js emits a plain static directory (`out/`) and does not
generate `server.js` or any server runtime. Running `next start` against a static-export build
fails immediately with `Could not find a production build`. The script will mislead any developer
who tries to preview a production build locally.
**Fix:** Replace `start` with a static file server preview command, or drop it entirely. The
Cloudflare Pages deployment does not invoke `next start`, and no CI config introduced in this
phase calls it:

```json
"scripts": {
  "dev": "next dev",
  "build": "next build",
  "preview": "npx serve out",
  "lint": "biome check",
  "format": "biome format --write"
}
```

`serve` can be added as a devDependency or called via `npx`. If no local preview workflow is
needed at this stage, simply remove the `start` script.

### WR-02: `globals.css` `body` rule overwrites Geist font variables with a static fallback

**File:** `web/app/globals.css:25`
**Issue:** `layout.tsx` loads Geist and Geist Mono via `next/font/google` and injects
`--font-geist-sans` / `--font-geist-mono` CSS custom properties onto `<html>`. `globals.css`
maps these into the Tailwind v4 `@theme` layer correctly:

```css
/* globals.css lines 11-12 — correct */
--font-sans: var(--font-geist-sans);
--font-mono: var(--font-geist-mono);
```

However, the `body` rule immediately below applies a hardcoded `font-family: Arial, Helvetica,
sans-serif;` (line 25), which takes precedence over any Tailwind `font-sans` utility class that
would use `--font-sans`. The result is that Geist never renders in the browser — Arial is served
instead — despite the font being fetched from Google Fonts on every page load.

```css
/* current — defeats Geist */
body {
  background: var(--background);
  color: var(--foreground);
  font-family: Arial, Helvetica, sans-serif;  /* overrides --font-sans */
}
```

**Fix:** Remove the hardcoded `font-family` and reference the CSS variable instead (with the
system-font stack as a true fallback), or use the Tailwind `font-sans` class on `<body>` in
`layout.tsx`:

```css
/* Option A — inline CSS variable */
body {
  background: var(--background);
  color: var(--foreground);
  font-family: var(--font-geist-sans), Arial, Helvetica, sans-serif;
}
```

```tsx
/* Option B — Tailwind class on <body> in layout.tsx */
<body className="min-h-full flex flex-col font-sans">{children}</body>
```

Either approach correctly activates the loaded Geist font.

### IN-02: Mixed version-pinning discipline in `web/package.json` devDependencies

**File:** `web/package.json:19-26`
**Issue:** Production dependencies (`next`, `react`, `react-dom`) and `@biomejs/biome` use exact
version pins. Dev dependencies use caret ranges (`"tailwindcss": "^4"`, `"typescript": "^5"`,
`"@types/node": "^20"`, `"@types/react": "^19"`, `"@types/react-dom": "^19"`,
`"@tailwindcss/postcss": "^4"`). Because `pnpm-lock.yaml` is committed the effective installed
versions are locked today. However, when `pnpm update` or a renovate/Dependabot run lands, caret
ranges will allow silent minor-version bumps for dev deps while prod deps stay frozen — creating
an asymmetric update model and potential for tooling surprises (e.g. a future Tailwind v4 minor
with breaking utility changes).
**Fix:** Pin devDependencies to exact versions to match the discipline used for production deps,
or explicitly document the intentional two-tier strategy:

```json
"@tailwindcss/postcss": "4.3.0",
"@types/node": "20.19.42",
"@types/react": "19.2.17",
"@types/react-dom": "19.2.3",
"tailwindcss": "4.3.0",
"typescript": "5.9.3"
```

Low urgency given the committed lockfile; address before adding a dependency-update bot.
