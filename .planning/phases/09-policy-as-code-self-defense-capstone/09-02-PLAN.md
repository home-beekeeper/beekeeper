<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI/shell environment → LoadLayered | BEEKEEPER_* env vars are attacker-influenceable in a compromised CI; crossed in applyEnvVars |
| project repo → LoadLayered | A repo's .beekeeper/config.json can relax user config (e.g. fail_mode: open); crossed in the project layer merge |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-05 | Tampering | applyEnvVars (env var injection mapping to arbitrary fields) | mitigate | Only a hardcoded known set of BEEKEEPER_* vars map to known fields; unknown vars are ignored, never reflectively applied |
| T-09-06 | Elevation of Privilege | project config relaxing security (fail_mode: open) | accept | By design — a project can relax user config (PRD §9); documented explicitly in config comments and THREAT-MODEL.md (Plan 05). The developer controls which repos they trust |
| T-09-07 | Tampering | merge resetting a secure field to zero value | mitigate | merge applies src only when non-zero (Pitfall 5); a partial higher layer cannot silently downgrade FailMode to the empty default |
| T-09-08 | Spoofing | invalid merged FailMode silently degrading security | mitigate | validate() reuses the Load FailMode switch; an invalid merged fail_mode returns an error rather than defaulting to a weaker mode |
</threat_model>
