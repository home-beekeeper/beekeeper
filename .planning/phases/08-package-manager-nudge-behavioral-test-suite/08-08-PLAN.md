<objective>
Add the `beekeeper nudge` CLI surface and make `beekeeper config set nudge.*` audit logging definite —
split out of Plan 06 so the CLI owns `cmd/beekeeper/*` exclusively (no files_modified overlap with
Plan 06's check/gateway/shim wiring; both are Wave 4 and run in parallel):

1. **`beekeeper nudge status|check|audit`** (NUDGE-07 / SC5) — thin Cobra `status|check|audit`
   subcommands mirroring `newPolicyCmd`: `status` prints human-readable PM state + active config,
   `check "<command>"` dry-runs pkgparse.Parse → nudge.DetectState → nudge.Evaluate and prints the
   decision/reason/rewritten, `audit [--since=...]` reuses audit.Query filtered to record_type "nudge".
2. **`beekeeper config set nudge.<key> <value>` audit logging** (WARNING 1 / §10-17) — PRD §5.2
   requires every config change be logged. There is NO existing `config set` command in the repo
   (verified — only the layered LoadLayered resolver in config_resolve.go). So this plan ADDS a thin
   `config set` subcommand that (a) loads the user config, (b) sets the nudge key, (c) saves it via
   config.Save, and (d) emits a config-change audit record. A test drives `config set nudge.mode hard`
   and asserts the audit record is written. §10-17 is a must-pass SC3 (BTEST-01) test — this makes it
   definite, not conditional.

Purpose: Keep Plan 06 within the task cap (it grew with the drift-check + advisory-cap tasks) and give
the CLI its own exclusive file ownership. All logic stays in internal/* (CLAUDE.md thin-wiring rule);
the §10-17 audit record uses the existing audit writer.
Output: `cmd/beekeeper/{nudge.go, config.go}` + `main.go` registration + their tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI arg → nudge check | Operator-supplied command string is dry-run parsed |
| CLI arg → config set | Operator-supplied config key/value is validated then persisted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-26 | Tampering (silent config change) | config set nudge.* | mitigate | Every `config set nudge.*` change is logged to the audit trail (§10-17, PRD §5.2) and validated via validateNudgeConfig before write (an invalid value is rejected fail-closed, no partial write) — Beekeeper MUST NOT silently change mode between sessions (PRD §5.2). |
| T-08-27 | Tampering (dry-run check exec) | nudge check | mitigate | `nudge check "<cmd>"` runs pkgparse.Parse (string-only) + nudge.DetectState (fixed-argv `--version` exec only); the operator's command string is NEVER passed to a shell — it is parsed for the dry-run decision, never executed. |
| T-08-28 | Information disclosure (audit query exposure) | nudge audit | accept | `nudge audit` reads the local audit NDJSON the operator already owns; no new exposure beyond the existing `beekeeper audit` query (reuses audit.Query). |
</threat_model>
