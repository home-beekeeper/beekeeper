<objective>
Add the `beekeeper nudge` CLI surface, make `beekeeper config set nudge.*` audit logging definite,
wire the gateway daemon's nudge config, and ship the PRD §13 operator docs — split out of Plan 06 so
the CLI owns `cmd/beekeeper/*` + `docs/nudge.md` exclusively (no files_modified overlap with Plan 06,
which now owns `internal/*` only). Plan 08 runs in Wave 5 (after Plan 06) because `main.go` references
`gateway.Config.Nudge` (defined by Plan 06) and `config.go` calls `config.ValidateNudgeConfig`
(exported by Plan 05) — both are directed cross-package compile dependencies, so file-ownership
disjointness alone is not sufficient. Wave 5 also holds Plan 07 (internal/check/* only); Plan 07 and
Plan 08 have ZERO files_modified overlap, so they remain parallel-safe within Wave 5:

1. **`beekeeper nudge status|check|audit`** (NUDGE-07 / SC5) — thin Cobra `status|check|audit`
   subcommands mirroring `newPolicyCmd`: `status` prints human-readable PM state + active config,
   `check "<command>"` dry-runs pkgparse.Parse → nudge.DetectState → nudge.Evaluate and prints the
   decision/reason/rewritten, `audit [--since=...]` reuses audit.Query filtered to record_type "nudge".
2. **`beekeeper config set nudge.<key> <value>` audit logging** (§10-17) — PRD §5.2 requires every
   config change be logged. There is NO existing `config set` command in the repo (verified — only
   the layered LoadLayered resolver in config_resolve.go). So this plan ADDS a thin `config set`
   subcommand that (a) loads the user config, (b) sets the nudge key, (c) validates the resulting
   config via the EXPORTED `config.ValidateNudgeConfig` (Plan 05) — rejecting an invalid value with a
   non-nil error and NO write (fail-closed §10-17; `cmd/beekeeper` is package `main` and cannot call
   an unexported validator, so Plan 05 exports it), (d) saves it via config.Save, and (e) emits a
   config-change audit record. A test drives `config set nudge.mode hard` and asserts the audit record
   is written, plus a rejection case `config set nudge.mode aggressive` asserting a non-nil error and
   NO write. §10-17 is a must-pass SC3 (BTEST-01) test — this makes it definite.
3. **Gateway daemon nudge wiring** (WARNING 3 daemon-side) — Plan 06 added the `Nudge nudge.Config`
   field to `gateway.Config` and a `nudge.DefaultConfig()` fallback in `newGatewayHandler`. The DAEMON
   that builds the `gateway.Config{...}` literal lives in THIS file (`cmd/beekeeper/main.go`
   `newGatewayCmd`, the literal at ~line 1220, where the layered `cfg.Nudge` is in scope), so this
   plan fills it: `gatewayCfg.Nudge = nudge.ConfigFrom(cfg.Nudge.Enabled, cfg.Nudge.Mode, ...)`.
   Because this line names the `gateway.Config.Nudge` field defined by Plan 06, this plan depends_on
   06 and runs in Wave 5 (a field defined in one plan and consumed in another is a directed compile
   dependency — file disjointness is necessary but not sufficient). This keeps the gateway-config edit
   on the CLI side (Plan 08 already owns main.go) so Plan 06 never touches cmd/beekeeper.
4. **`docs/nudge.md` + commented default config block** (PRD §13 / WARNING 5) — the operator doc
   that PRD §13 mandates: feature explanation, version floors, the soft-vs-hard distinction, and the
   Node 22 (Maintenance-LTS) caveat for pnpm 11; plus a default nudge config block with inline
   comments explaining each field. No plan produced it before this fix.

Purpose: Keep Plan 06 within the task cap and confined to internal/*, and give the CLI its own
exclusive file ownership (cmd/beekeeper/* + docs/nudge.md). All logic stays in internal/* (CLAUDE.md
thin-wiring rule); the §10-17 audit record uses the existing audit writer; the gateway nudge config
goes through the single nudge.ConfigFrom mapper; the config validation goes through the single
EXPORTED config.ValidateNudgeConfig.

+ `docs/nudge.md` + their tests.
