<objective>
Wire the nudge into the three live enforcement consumers and add the gateway drift check:

1. **check hook** (`runCheck` + new `nudge_adapter.go`) — parse the Bash command via
   `pkgparse.Parse`; when `IsInstall`, resolve `PMState` FRESH via `nudge.DetectState` (NO cache —
   Flag 2 Position B, the one-shot hook gets no cache hits), call the pure `nudge.Evaluate`, map
   the nudge `Decision` to a `policy.Decision`, and `mergeDecisions` it AFTER `ApplyPolicyOverlay`
   (CR-02 ordering — an allow overlay must not downgrade a nudge Block). Emit a `record_type:"nudge"`
   audit record carrying the §9 fields AND `nudge_action` (= nudge.ActionString).
2. **gateway** (`applyPolicy` + `proxy.go`) — same merge, but this is the ONLY place the 60s
   `nudge.Cache` is constructed (once at start) and reused per request. The at-most-one-advisory-
   per-session cap (NUDGE-03, RESEARCH Pitfall 4 / Open Q3) is gateway-session state keyed by
   agent-id when present, else process-global.
3. **shim** (`shim.go`) — the npm shim calls `pkgparse.Parse` → `nudge.Evaluate` before proxying.
4. **gateway drift check** (`drift.go`, BLOCKER 1 / §10-15 / Open Q2) — a periodic (weekly, per
   cfg.Nudge.MajorDriftCheck.Interval) task in the long-lived gateway with an INJECTED metadata-
   fetch fn (returning e.g. "pnpm 12.0.0"); it calls `isMajorDrift`, and when a new major is found
   emits an ASYNC, non-blocking `record_type:"version_drift"` severity-info record. It NEVER runs on
   the check hot path and does NOT auto-update floors (PRD §7.1, Out-of-Scope).

The `beekeeper nudge` CLI surface (status|check|audit) + the §10-17 config-change audit logging are
split into Plan 08 (08-08) so this plan stays within the task cap and the CLI owns cmd/beekeeper/*
exclusively (no files_modified overlap with this plan — both are Wave 4, parallel-safe).

Purpose: This is the Phase 7 most-restrictive-merge pattern repeated for nudge. Soft mode
(Advise/Rewrite → Level "warn") keeps exit 0; only `Block` (requireHardened) is restrictive — so
the merge keeps `exitCodeFor` semantics intact.
Output: live nudge wiring in check/gateway/shim + the gateway drift check. (Behavioral tests proving
the wiring are Plan 07; the CLI is Plan 08.)
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent Bash command → runCheck / applyPolicy / shim | Attacker-influenced install command reaches each enforcement consumer |
| package registry metadata → gateway drift check | Untrusted version metadata is fetched (off the hot path) and compared to floors |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-17 | Tampering (overlay downgrade of a Block) | runCheck/applyPolicy merge ordering | mitigate | The nudge merge runs AFTER ApplyPolicyOverlay via mergeDecisions (most-restrictive-wins), so a package_allowlist allow cannot downgrade a requireHardened nudge Block (CR-02 ordering repeated). |
| T-08-18 | DoS (detection on the hot path) | check hook detection | mitigate | Detection runs ONLY when parsed.IsInstall (non-install commands never exec); the check hook uses fresh DetectState bounded at 2s (no cache I/O — Flag 2 Position B, Pitfall 2); the gateway uses the 60s cache so one slow call per minute, not per request. |
| T-08-19 | Repudiation (untraced nudge actions) | audit record emission | mitigate | Every nudge decision emits a record_type "nudge" record (original_command, rewritten_command, reason_code, pm_state, nudge_action) so forensic review can trace which agent command led to which PM invocation (PRD §12, §9). |
| T-08-24 | DoS (drift network call blows the budget) | gateway drift check | mitigate | The drift check is a gateway-owned periodic goroutine (weekly), NEVER on the per-request applyPolicy / check hot path (Pitfall 6); the metadata fetch is async + best-effort; a fetch error emits nothing and never blocks. |
| T-08-25 | Tampering (silent trust expansion via drift) | drift floor update | mitigate | The drift check emits an info-only version_drift record and does NOT auto-update floors (PRD §7.1, Out-of-Scope) — a new major requires explicit Beekeeper review, so a poisoned "new version" signal cannot silently widen the accepted version range. |
</threat_model>
