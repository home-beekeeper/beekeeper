<objective>
Wire the nudge into all three live enforcement consumers and add the CLI surface:

1. **check hook** (`runCheck` + new `nudge_adapter.go`) — parse the Bash command via
   `pkgparse.Parse`; when `IsInstall`, resolve `PMState` FRESH via `nudge.DetectState` (NO cache —
   Flag 2 Position B, the one-shot hook gets no cache hits), call the pure `nudge.Evaluate`, map
   the nudge `Decision` to a `policy.Decision`, and `mergeDecisions` it AFTER `ApplyPolicyOverlay`
   (CR-02 ordering — an allow overlay must not downgrade a nudge Block). Emit a `record_type:"nudge"`
   audit record.
2. **gateway** (`applyPolicy`) — same merge, but this is the ONLY place the 60s `nudge.Cache` is
   constructed (once at start) and reused per request.
3. **shim** (`shim.go`) — the npm shim calls `pkgparse.Parse` → `nudge.Evaluate` before proxying.
4. **CLI** (`cmd/beekeeper/nudge.go`) — thin Cobra `status|check|audit` subcommands (NUDGE-07).

Purpose: This is the Phase 7 most-restrictive-merge pattern repeated for nudge. Soft mode
(Advise/Rewrite → Level "warn") keeps exit 0; only `Block` (requireHardened) is restrictive — so
the merge keeps `exitCodeFor` semantics intact.
Output: live nudge wiring in check/gateway/shim + the `beekeeper nudge` command. (Behavioral tests
proving the wiring are Plan 07.)
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent Bash command → runCheck / applyPolicy / shim | Attacker-influenced install command reaches each enforcement consumer |
| CLI arg → nudge check | Operator-supplied command string is dry-run parsed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-17 | Tampering (overlay downgrade of a Block) | runCheck/applyPolicy merge ordering | mitigate | The nudge merge runs AFTER ApplyPolicyOverlay via mergeDecisions (most-restrictive-wins), so a package_allowlist allow cannot downgrade a requireHardened nudge Block (CR-02 ordering repeated). |
| T-08-18 | DoS (detection on the hot path) | check hook detection | mitigate | Detection runs ONLY when parsed.IsInstall (non-install commands never exec); the check hook uses fresh DetectState bounded at 2s (no cache I/O — Flag 2 Position B, Pitfall 2); the gateway uses the 60s cache so one slow call per minute, not per request. |
| T-08-19 | Repudiation (untraced nudge actions) | audit record emission | mitigate | Every nudge decision emits a record_type "nudge" record (original_command, rewritten_command, reason_code, pm_state) so forensic review can trace which agent command led to which PM invocation (PRD §12). |
| T-08-20 | Tampering (config silent change) | config set nudge.* | mitigate | Config changes to nudge keys are logged to the audit trail (§10-17); Beekeeper MUST NOT silently change mode between sessions (PRD §5.2). |
