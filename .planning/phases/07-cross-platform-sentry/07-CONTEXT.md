# Phase 7: Cross-Platform Sentry — Context

**Gathered:** 2026-05-28
**Status:** Ready for planning
**Source:** PRD Express Path (beekeeper-prd.md)

<domain>
## Phase Boundary

Phase 7 delivers the cross-platform extension of the Sentry daemon (Phase 5 built Linux):

- **macOS Sentry** via `eslogger` subprocess — no EndpointSecurity entitlement required for v1; requires `sudo`; launchd plist installation; same 5-rule correlation engine and 7-day baseline as Linux
- **Windows Sentry** via ETW using `tekert/golang-etw` (no CGO) — Windows Service under LocalService privileges; NT Kernel Logger conflict detection; EventsLost surfacing in `beekeeper diag`; named pipe IPC with DACL
- **SLSA Level 3 provenance** — `slsa-github-generator@v2.1.0` (full semver, NOT `@v2`) integrated into GoReleaser pipeline; CycloneDX SBOM generated by `syft`
- **Windows named pipe IPC** replacing Unix socket for CLI↔Sentry on Windows; DACL grants access only to the installing user's SID

This phase does NOT include:
- macOS EndpointSecurity entitlement (v2 path, non-trivial for indie OSS)
- TUI dashboard (Phase 8)
- Policy as code / beekeeper-self catalog (Phase 9)

</domain>

<decisions>
## Implementation Decisions

All decisions below are extracted from the PRD and are LOCKED. They are treated as acceptance criteria, not suggestions.

### macOS Sentry: eslogger-based (SMAC-01 to SMAC-04)

- `beekeeper protect install` on macOS installs a launchd plist (privileged, requires `sudo`)
- `beekeeper protect uninstall` and `beekeeper protect status` work on macOS
- eslogger subprocess started via `exec.CommandContext`; stdout pipe drained by a high-priority goroutine
- eslogger output is NDJSON; events are parsed and normalized to the common `SentryEvent` struct already defined in Phase 5
- The same 5 default Sentry rules apply to normalized macOS events; the 7-day audit-only baseline period applies identically
- `beekeeper protect status` explicitly documents eslogger coverage gaps:
  - Keychain / Security framework API access: **not observable** by eslogger
  - In-memory Cocoa API operations: **not observable** by eslogger
- eslogger field names are **partially undocumented** — the parser must be built against real `eslogger` output on `macos-latest` CI, not synthetic JSON. This is a hard constraint per CLAUDE.md research note.

### Windows Sentry: ETW-based (SWIN-01 to SWIN-06)

- `beekeeper protect install` on Windows installs a Windows Service running under LocalService privileges
- `beekeeper protect uninstall` and `beekeeper protect status` work on Windows
- ETW event ingestion uses `tekert/golang-etw` library (**no CGO** — NOT `bi-zone/etw` which requires CGO). This is a locked architectural constraint from CLAUDE.md.
- ETW providers: `Microsoft-Windows-Kernel-Process` (process events), `Microsoft-Windows-Security-Auditing` (file/network events)
- At install time, Beekeeper probes for an existing NT Kernel Logger session conflict (only one session allowed per machine); if an EDR already owns the session, this is surfaced in `beekeeper protect status`
- `EventsLost` counter from ETW is surfaced in `beekeeper diag` output so users know when detection coverage is degraded under load
- The same 5 default Sentry rules and 7-day baseline apply to normalized ETW events
- IPC between the unprivileged CLI and the Sentry Windows Service uses a Windows named pipe with a DACL that grants access only to the installing user's SID (not just LocalService)

### Windows Named Pipe IPC (SWIN-05)

- Phase 5 built Unix socket IPC (`internal/ipc/`) with a Windows stub returning `ErrNotSupported`
- Phase 7 replaces that stub with a real Windows named pipe implementation
- DACL is set at pipe creation to grant GENERIC_READ|GENERIC_WRITE to the installing user's SID only
- Protocol framing is the same as Unix: 4-byte big-endian length prefix + JSON payload
- Max message size cap: 64KB (same as Unix IPC)

### SLSA Level 3 + SBOM (SFDF-05)

- `slsa-github-generator@v2.1.0` — FULL semver pinned, NOT `@v2`. This is a locked constraint from CLAUDE.md.
- SBOM generated by `syft` in CycloneDX format and published with each release
- Both provenance attestation and SBOM are wired into the GoReleaser pipeline (existing `.goreleaser.yml` from Phase 1)
- `make verify-release VERSION=X.Y.Z` must remain functional after these changes
- Sigstore signing from Phase 1 is preserved/not broken

### Common Sentry event normalization

- macOS and Windows events normalize to the same `SentryEvent` struct defined in Phase 5 (`internal/sentry/event.go` or equivalent)
- The correlation rule engine from Phase 5 runs unchanged on macOS and Windows normalized events
- Platform-specific ingestion packages: `internal/sentry/darwin/` and `internal/sentry/windows/`
- Existing `internal/sentry/linux/` package from Phase 5 is not modified in this phase

### Failure modes

- eslogger crash on macOS: supervised via `exec.CommandContext`; logged; Sentry status degrades gracefully
- ETW session unavailable on Windows (conflict or rate limit): EventsLost surfaced; detection degrades, not panics
- Sentry daemon crash on either platform: supervisor restarts with exponential backoff (same behavior as Linux)
- IPC named pipe unavailable: CLI commands return structured error

### Claude's Discretion

- Exact launchd plist label (e.g., `com.mzansi.beekeeper.sentry`)
- Exact Windows Service name string
- Named pipe path (e.g., `\\.\pipe\beekeeper-sentry`)
- How to probe NT Kernel Logger session (Windows ETW session enumeration API)
- Order of ETW provider registration vs. session startup
- Test fixtures for synthetic eslogger NDJSON (since real eslogger output only available in CI)
- How to unit-test Windows Service registration without requiring admin rights on dev machine
- ETW buffer sizing defaults pending empirical measurement (flagged in CLAUDE.md research note)

</decisions>

<specifics>
## Specific Ideas

### From PRD §4.1 (Sentry daemon)

> Implementation: fanotify + eBPF on Linux; **eslogger on macOS** (which streams EndpointSecurity events to processes running as root without requiring the entitlement); **ETW with the relevant security providers on Windows**.

### From PRD §8.4 (Elevation strategy)

> *macOS.* EndpointSecurity framework requires Apple's entitlement. Beekeeper v1 ships with `eslogger`-based monitoring on macOS: works, requires `sudo`, no entitlement needed, less complete than full EndpointSecurity but covers the core attack patterns.

> *Windows.* The Sentry daemon installs as a Windows Service running with SYSTEM or LocalService privileges. ETW also rate-limits under load; Beekeeper surfaces this in `beekeeper diag` so users know when detection coverage is degraded.

### From PRD §12.4 (IPC security)

> IPC authorization on macOS (Unix permissions + XPC entitlement) and Windows (named pipe ACLs). Two-account release approval rule active in GitHub repo settings. Verification path documented in `BUILDING.md` with `make verify-release` target. Cross-platform reproducible builds verified in CI matrix.

### From PRD §12.2 (SLSA)

> SLSA Level 3 provenance as the v0.9.0 target. Every binary ships with attestation describing which workflow built it, which commit, which dependencies.

### From CLAUDE.md

> **ETW: `tekert/golang-etw` (no CGO)** — NOT `bi-zone/etw` which requires CGO.
> **SLSA: Level 3 via `slsa-github-generator@v2.1.0` (full semver — NOT `@v2`)**
> **Phase 7 research note**: eslogger field names partially undocumented — parse against real `macos-latest` output; ETW buffer sizing needs measurement under load

</specifics>
