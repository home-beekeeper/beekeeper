## Summary

Phase 7 adds cross-platform Sentry coverage (macOS via eslogger subprocess, Windows via ETW) on top of the Linux Sentry built in Phase 5. It also ships SLSA Level 3 provenance and a CycloneDX SBOM for every release. All five correlation rules and the 7-day baseline already exist in `internal/sentry/` with no build tags — they compile everywhere and need no modification.

The main implementation surface is four new packages plus goreleaser pipeline additions:

1. `internal/sentry/darwin/` — eslogger subprocess supervisor, NDJSON parser, normalization, launchd install/uninstall
2. `internal/sentry/windows/` — ETW ingestion via `tekert/golang-etw`, normalization, NT Kernel Logger conflict probe, Windows Service install/uninstall
3. `internal/ipc/pipe_windows.go` — Replace `stub.go` with a real Windows named pipe implementation with DACL
4. `.goreleaser.yaml` + `.github/workflows/release.yml` additions — SBOM (syft) and SLSA Level 3

The most significant risk area is Windows ETW: the `tekert/golang-etw` library (v0.6.2) has sparse pkg.go.dev documentation. Key API signatures (Session constructor, AddProvider, EventsLost field location) were not fully extractable from public docs and are marked [ASSUMED]. A brief direct code inspection in CI before implementation is recommended.

The second risk area is eslogger: JSON field names are partially undocumented by Apple. CLAUDE.md explicitly requires the parser be built and validated against real `macos-latest` CI output. The planned approach is: create synthetic stubs for unit tests, validate field names in a dedicated CI job on `macos-latest` that runs `sudo eslogger exec create --format json | head -5` and compares against the parser, failing loudly if fields don't match.

**Primary recommendation:** Use `microsoft/go-winio` for the Windows named pipe implementation — it already handles DACL, SDDL security descriptors, and message framing correctly, and is well-tested in Docker/containerd/WireGuard. Avoid reimplementing raw `CreateNamedPipe` + `SetSecurityDescriptor` from scratch. For ETW, use `tekert/golang-etw` v0.6.2 as locked by CLAUDE.md and confirm the Session/Consumer API by reading source directly in the CI step that adds it to go.mod.

## 2. macOS Sentry: eslogger

### 2.1 eslogger Invocation

[VERIFIED: keith.github.io/xcode-man-pages/eslogger.1.html]

eslogger is included in macOS since Ventura (13.0). It streams EndpointSecurity events as NDJSON to stdout. It requires root (sudo) and Full Disk Access TCC authorization.

```bash
# Run as root, stream exec + create + network events
sudo eslogger exec create network_flow
```

Key events for Beekeeper Sentry:

| Event Name | What it covers |
|-----------|----------------|
| `exec` | Process execution (execve equivalent) — includes new process path, args, parent |
| `create` | File creation events |
| `open` | File open events (for credential file access) |
| `network_flow` | Outbound network connections |
| `fork` | Process fork (useful for tree reconstruction) |

**Note:** Use `eslogger --list-events` to get the current full list on a given macOS version. The list can vary across OS versions. [VERIFIED: man page]

**Critical constraint from CLAUDE.md:** eslogger field names are partially undocumented by Apple. The parser MUST be validated against real `macos-latest` CI output, not synthetic JSON. A dedicated CI step must run `sudo eslogger exec --format json | head -3` and compare field names against the parser struct.

### 2.3 eslogger Subprocess Supervisor Pattern

[VERIFIED: exec.CommandContext standard library docs, go.dev/pkg/os/exec]

```go
// internal/sentry/darwin/daemon.go
//go:build darwin

func RunDaemon(ctx context.Context, cfg *config.Config, auditPath string) error {
    cmd := exec.CommandContext(ctx, "eslogger", "exec", "open", "create", "network_flow", "fork")
    cmd.Stdout = nil // we set a pipe below
    cmd.Stderr = os.Stderr

    stdout, err := cmd.StdoutPipe()
    if err != nil {
        return fmt.Errorf("eslogger stdout pipe: %w", err)
    }

    if err := cmd.Start(); err != nil {
        return fmt.Errorf("eslogger start: %w", err)
    }

    // High-priority goroutine drains stdout — must not block
    events := make(chan sentry.SentryEvent, 10000)
    go drainEslogger(ctx, stdout, events)

    // Start IPC server and correlation engine...
    // ...

    err = cmd.Wait()
    // cmd.Wait() returns when ctx is cancelled (exec.CommandContext sends SIGKILL)
    return err
}

func drainEslogger(ctx context.Context, r io.Reader, ch chan<- sentry.SentryEvent) {
    scanner := bufio.NewScanner(r)
    scanner.Buffer(make([]byte, 64*1024), 64*1024)
    for scanner.Scan() {
        line := scanner.Bytes()
        ev, err := parseEsloggerLine(line)
        if err != nil {
            // skip malformed lines; log at debug level
            continue
        }
        select {
        case ch <- ev:
        default:
            // channel full: drop event, increment dropped counter
        }
    }
}
```

**Key points:**
- `exec.CommandContext` sends SIGKILL when ctx is cancelled — eslogger is killed cleanly
- High-priority goroutine: must never block; use non-blocking channel send with `select`/`default`
- `bufio.Scanner` with enlarged buffer handles long JSON lines (eslogger events can be large)
- eslogger does NOT require any special entitlement when run as root — this is the v1 mechanism [VERIFIED: CLAUDE.md, PRD §8.4]

// Attach to sessions by name (after creating real-time sessions separately)

// OR attach to Session objects:

// Register event callback
c.ProcessEvents(func(e *etw.Event) {

### 3.6 NT Kernel Logger Session Conflict Probe (SWIN-03)

There is exactly one "NT Kernel Logger" session allowed per machine (name: `"NT Kernel Logger"`). If an EDR already holds it, attempting to create a second session with kernel providers will fail. [VERIFIED: Microsoft ETW docs]

Probe pattern:

```go
// Try to open (not create) the NT Kernel Logger session
// If it succeeds, it means an EDR owns it already
func probeNTKernelLoggerConflict() bool {
    // Attempt a real-time trace with the kernel logger name
    // If OpenTrace returns ERROR_ALREADY_EXISTS or the session exists,
    // report conflict
    sess, err := etw.NewRealTimeSession("NT Kernel Logger")
    if err != nil {
        // err may indicate session already owned by another process
        // Check for specific error code (ERROR_ALREADY_EXISTS = 183)
        return isAlreadyExistsError(err)
    }
    // We got it — this is unexpected on a normal system with an EDR
    sess.Close()
    return false
}
```

**Alternative:** Use `etw.GetSessionNames()` or equivalent to enumerate existing sessions before attempting to create. [ASSUMED — API availability unknown]

**Fallback:** If the ETW creation call returns an error, check `windows.ERROR_ALREADY_EXISTS` (system error 183) and surface a clear message in `beekeeper protect status`.

## 4. Windows Service Installation (SWIN-01)

### 4.1 mgr.CreateService API

[VERIFIED: pkg.go.dev/golang.org/x/sys/windows/svc/mgr]

```go
import (
    "golang.org/x/sys/windows/svc/mgr"
    "golang.org/x/sys/windows"
)

func InstallWindowsService(exePath string) error {
    m, err := mgr.Connect()
    if err != nil {
        return fmt.Errorf("connect to SCM: %w", err)
    }
    defer m.Disconnect()

    s, err := m.CreateService("BeekeeperSentry", exePath, mgr.Config{
        ServiceType:      windows.SERVICE_WIN32_OWN_PROCESS,
        StartType:        mgr.StartAutomatic,
        ErrorControl:     mgr.ErrorNormal,
        ServiceStartName: "NT AUTHORITY\\LocalService",
        DisplayName:      "Beekeeper Sentry Daemon",
        Description:      "Beekeeper OS-level security monitoring (ETW event ingestion)",
    })
    if err != nil {
        return fmt.Errorf("create service: %w", err)
    }
    defer s.Close()
    return s.Start()
}
```

### 4.2 LocalService Privileges

`NT AUTHORITY\LocalService` [VERIFIED: pkg.go.dev/golang.org/x/sys/windows/svc/mgr — ServiceStartName field]:
- Has limited local privileges
- Accesses network as an anonymous user
- No password required (`Config.Password` left empty)
- Requires `windows.SERVICE_WIN32_OWN_PROCESS` for `ServiceType`

**LocalService for ETW (open question):** ETW kernel provider subscriptions typically require SeSecurityPrivilege or membership in "Event Log Readers" group. LocalService may lack this. If kernel-level providers require higher privilege, the Windows Sentry may need to run as SYSTEM or NetworkService. [ASSUMED — needs live Windows CI validation]

### 4.3 Service Run Loop Pattern

```go
//go:build windows

package main

import (
    "golang.org/x/sys/windows/svc"
)

type beekeeperSentryService struct{}

func (s *beekeeperSentryService) Execute(args []string, r <-chan svc.ChangeRequest, status chan<- svc.Status) (bool, uint32) {
    status <- svc.Status{State: svc.Running, Accepts: svc.AcceptStop | svc.AcceptShutdown}

    // Start ETW ingestion goroutine...

    for {
        select {
        case req := <-r:
            switch req.Cmd {
            case svc.Stop, svc.Shutdown:
                // Cancel context, wait for goroutines
                status <- svc.Status{State: svc.StopPending}
                return false, 0
            }
        }
    }
}

// Entry point when running as service:
func runAsService() error {
    return svc.Run("BeekeeperSentry", &beekeeperSentryService{})
}
```

**Option A: `microsoft/go-winio` (RECOMMENDED)**
- `ListenPipe(path string, c *PipeConfig) (net.Listener, error)` [VERIFIED: github.com/microsoft/go-winio]
- `PipeConfig.SecurityDescriptor` accepts SDDL string — cleanest way to express DACL
- Returns `net.Listener` — integrates directly with existing `internal/ipc/server.go` Accept loop
- SDDL for "installing user only": `D:(A;;GRGW;;;{user-SID})`
- Used in production by Docker, containerd, WireGuard — well-tested on Windows
- Adds one dependency: `github.com/microsoft/go-winio`

**Option B: Raw `golang.org/x/sys/windows.CreateNamedPipe`**
- `windows.CreateNamedPipe(name *uint16, openMode, pipeMode, maxInstances, outBufSize, inBufSize, defaultTimeout uint32, sa *windows.SecurityAttributes) (handle windows.Handle, err error)` [ASSUMED — signature from pkg.go.dev partial docs]
- Requires building `SecurityAttributes` manually with a custom DACL
- More code, more opportunity for bugs, no existing test coverage

**Recommendation:** Use `microsoft/go-winio` because:
1. It returns `net.Listener` which the existing IPC server pattern (`internal/ipc/server.go`) can reuse directly on the server side
2. SDDL strings are well-documented and easier to audit than manual ACL construction
3. `pipe_windows.go` will be substantially simpler

### 5.3 Named Pipe with go-winio

```go
//go:build windows

package ipc

import (
    "context"
    "fmt"
    "net"
    "time"

    "github.com/microsoft/go-winio"
    "golang.org/x/sys/windows"
)

// pipePath is the canonical named pipe path for Beekeeper Sentry IPC.
const pipePath = `\\.\pipe\beekeeper-sentry`

func NewServer(sockPath string, ownerUID uint32) (*Server, error) {
    // Build SDDL: Discretionary ACL granting GENERIC_READ|GENERIC_WRITE to the
    // installing user only. The SDDL format:
    //   D:(A;;GRGW;;;{SID})
    // where GRGW = GENERIC_READ | GENERIC_WRITE
    sid, err := getCurrentUserSID()
    if err != nil {
        return nil, fmt.Errorf("get user SID: %w", err)
    }
    sddl := fmt.Sprintf("D:(A;;GRGW;;;%s)", sid.String())

    l, err := winio.ListenPipe(pipePath, &winio.PipeConfig{
        SecurityDescriptor: sddl,
        MessageMode:        false, // byte mode; framing handled by our protocol
        InputBufferSize:    65536,
        OutputBufferSize:   65536,
    })
    if err != nil {
        return nil, fmt.Errorf("listen pipe: %w", err)
    }
    return &Server{listener: l}, nil
}

// Connect (client side) — replaces stub
func Connect(sockPath string, timeout time.Duration) (net.Conn, error) {
    ctx, cancel := context.WithTimeout(context.Background(), timeout)
    defer cancel()
    return winio.DialPipeContext(ctx, pipePath)
}
```

**Note on ownerUID:** On Windows there is no UID in the Unix sense. The `ownerUID uint32` parameter in the existing `NewServer` signature should be preserved for API compatibility but can be unused on Windows (the SID is obtained from the process token instead).

### 5.4 Protocol Framing

The existing `Encode`/`Decode` functions in `internal/ipc/proto.go` (4-byte big-endian length prefix + JSON, 64KB cap) are reused without modification. `pipe_windows.go` implements the same `Server`/`Connect` interface as `server.go`/`client.go` but uses a named pipe instead of a Unix socket. [VERIFIED: 05-01-PLAN.md — framing design]

## 7. Common Pitfalls

### Pitfall 1: eslogger Requires Root — Missing sudo in launchd Plist
**What goes wrong:** eslogger exits immediately with "eslogger must be run as super-user" if the daemon process isn't root.
**Why it happens:** launchd plist without `<key>UserName</key><string>root</string>` runs as the installing user.
**How to avoid:** Ensure plist has `UserName: root`. [VERIFIED: eslogger man page]

### Pitfall 2: eslogger JSON Field Names Change Across macOS Versions
**What goes wrong:** Parser silently produces empty SentryEvent fields — rules never fire.
**Why it happens:** Apple provides no stable schema guarantee; `schema_version` increments with OS updates.
**How to avoid:** Mandatory CI job on `macos-latest` that validates parser fields against live eslogger output. Add explicit field presence checks with test assertions.

### Pitfall 3: tekert/golang-etw vs tekert/goetw — Wrong Import Path
**What goes wrong:** Build fails or uses wrong library version.
**Why it happens:** `tekert/goetw` is a newer fork at v0.8.9; `tekert/golang-etw` is the one locked by CLAUDE.md.
**How to avoid:** Always import `github.com/tekert/golang-etw/etw` at v0.6.2. Do not upgrade to goetw without explicit approval.

### Pitfall 4: ETW Security-Auditing Provider Inaccessible from LocalService
**What goes wrong:** `AddProvider` for Security-Auditing returns ACCESS_DENIED from LocalService.
**Why it happens:** `Microsoft-Windows-Security-Auditing` requires elevated privileges (SYSTEM or SeSecurityPrivilege).
**How to avoid:** Build fallback — if Security-Auditing fails, use `Microsoft-Windows-Kernel-File` and `Microsoft-Windows-Kernel-Network` providers instead. Surface the degradation in `beekeeper protect status`.
**Warning signs:** `AddProvider` returns error 5 (ERROR_ACCESS_DENIED) at service startup.

### Pitfall 5: NT Kernel Logger Session Conflict — Not Checking Before Install
**What goes wrong:** `beekeeper protect install` returns a confusing error; user sees "session already exists".
**Why it happens:** EDR software (Windows Defender, CrowdStrike, etc.) often holds the NT Kernel Logger session.
**How to avoid:** Probe at install time (SWIN-03). Surface as human-readable message: "An existing EDR session (NT Kernel Logger) was detected. ETW coverage may be limited to supplementary providers."

### Pitfall 6: Named Pipe Path Must Use `\\.\pipe\` Prefix
**What goes wrong:** `ListenPipe` or `CreateNamedPipe` fails with invalid parameter.
**Why it happens:** Named pipe paths on Windows require the canonical `\\.\pipe\<name>` format; `~/.beekeeper/sentry.sock` is a Unix path and invalid on Windows.
**How to avoid:** Always use `\\.\pipe\beekeeper-sentry` as the pipe path. Keep the `sockPath` parameter in IPC API signatures but ignore it on Windows, using the canonical constant instead.

### Pitfall 7: SLSA Generator `@v2` Instead of `@v2.1.0`
**What goes wrong:** SLSA verification fails; provenance attestation is rejected by slsa-verifier.
**Why it happens:** The SLSA framework requires full semver pinning for trusted builder verification.
**How to avoid:** Always use `slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@v2.1.0` — exact string as specified in CLAUDE.md. Never abbreviate to `@v2`. [VERIFIED: SLSA generator README, CLAUDE.md]

### Pitfall 8: syft Not Installed Before GoReleaser Runs
**What goes wrong:** GoReleaser exits with "syft: command not found" during SBOM generation.
**Why it happens:** `syft` is not pre-installed on `ubuntu-latest` GitHub runners.
**How to avoid:** Add `anchore/sbom-action/download-syft@v0` step before the GoReleaser step in release.yml.

### Pitfall 9: go-winio Named Pipe in Byte Mode — Framing Relies on Length Prefix
**What goes wrong:** `Decode` blocks or receives partial messages.
**Why it happens:** Named pipes in byte mode (not message mode) do not preserve message boundaries. The existing IPC framing (4-byte length prefix + io.ReadFull) handles this correctly — do not switch to message mode which has a 64KB limit coinciding with Beekeeper's own cap but behaves differently.
**How to avoid:** Keep `MessageMode: false` in PipeConfig. The existing `Encode`/`Decode` framing already handles byte stream correctly with `io.ReadFull`. [VERIFIED: 05-01-PLAN.md framing design]

### Pitfall 10: eslogger `open` vs `create` for File Access Detection
**What goes wrong:** Credential file reads are not detected because only `create` is subscribed.
**Why it happens:** `open` is the event for file reads; `create` is for file creation. Credential theft reads `.ssh/id_rsa` — that's an `open`, not a `create`.
**How to avoid:** Subscribe to both `exec`, `open`, `create`, `network_flow`, `fork`. Map `open` events to `EventFileAccess` in SentryEvent.

## 11. Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Named pipe DACL restricts access to installing user SID |
| V3 Session Management | yes | 4-byte length prefix + 64KB cap; same as Unix IPC |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Unauthorized CLI→Sentry pipe access | Elevation of Privilege | go-winio SDDL DACL — only installing user's SID gets GENERIC_READ/WRITE |
| Malformed eslogger NDJSON injected via pipe hijack | Tampering | bufio.Scanner + JSON unmarshaling; malformed lines are dropped |
| ETW event flood causing consumer lag | Denial of Service | EventsLost surfaced; event channel cap; daemon does not block on full channel |
| SLSA generator `@v2` mutable tag | Supply Chain | Mandatory full semver `@v2.1.0` — verified by CI grep gate |
| Compromised syft in pipeline | Supply Chain | Pin syft via `anchore/sbom-action/download-syft@v0` (pinned action SHA in production) |
| EDR session conflict causes silent ETW failure | Denial of Service | Probe at install time (SWIN-03); surface conflict in status; degrade gracefully |
