<title>#3 SERIOUS: wire eBPF bytecode generation into the Linux build pipeline (SLNX-02/04)</title>
<read_first>
- internal/sentry/linux/gen.go (//go:generate bpf2go directives)

<action>
Do NOT generate or run eBPF bytecode locally (requires Linux + clang/llvm). Instead wire generation into the build pipeline so Linux release/CI builds embed real bytecode: (a) add a `make generate` target that runs `go generate ./internal/sentry/linux/...` (guarded to Linux); (b) add a CI step in the Linux job that runs `go generate ./internal/sentry/linux/...` (installing clang/llvm/libbpf as needed) BEFORE `go build`/tests, and fails the job if generation fails; (c) add a GoReleaser `before.hooks` entry (Linux build only) running the same generate. Keep the loader stubs fail-closed with a clear error when bytecode is absent. Add a short note to CLAUDE.md's eBPF section: bytecode is CI-generated at build time, never committed, never runtime-compiled. While editing .goreleaser.yaml, also add `-buildvcs=false` to the build `flags` list alongside `-trimpath` (fixes the minor reproducibility gap).
</action>
<acceptance_criteria>
- `make generate` target exists and runs `go generate ./internal/sentry/linux/...`
- CI Linux job runs `go generate ./internal/sentry/linux/...` before build; .goreleaser.yaml has a Linux before-hook for it and `-buildvcs=false` in flags
- CLAUDE.md eBPF section notes CI-generated-at-build-time bytecode
- Loader stub still returns a clear fail-closed error when bytecode absent (unchanged behavior on non-generated trees)
