<objective>
Land the existing-package prerequisites that the new `internal/corpus` package (Plan 02/03) and Phase 23 will build on, WITHOUT creating the corpus package itself. Three additive, non-breaking changes to existing packages:

1. Add `SourceSurface`, `ClusterID`, `RulesetVersion` to `audit.AuditRecord` (additive `omitempty` fields) — the four-layer schema's behavior/decision-layer slots that the embedded record must carry (SCHEMA-01, SCHEMA-02, SCHEMA-05). Document `"alert"` as the fourth valid `Decision` value (OQ-1 resolution; no type change).
2. Add `audit.RedactRecordWithDefaults(rec) AuditRecord` to clear the unexported-`redactPattern` landmine so Phase 23's corpus store can redact (research Finding 7).
3. Add `policy.CorroborateOutcome` (+ `CorroborationOutcome` struct) exported wrapper over the unexported `corroborate()` (research Finding 8), and `config.CorpusConfig` following the `AuditConfig` pattern (research Finding 10).

Purpose: These are the cross-package contracts that must exist before the corpus types compile and before Phase 23 can wire the store/adjudicator. Resolving them here keeps the corpus package pure and unblocks STORE-02/ADJ-01.

Output: Three modified production files + three extended test files, all additive and backward-compatible.

Resolved open questions (per planning context):
- OQ "source_surface + cluster_id placement" → on `audit.AuditRecord` (additive `omitempty`). Rationale: forensic value in the raw audit log too, and the Sentry surface can set `cluster_id` directly; non-breaking because both names are new and `omitempty`.
- OQ "verdict fourth value `alert`" → documented as a valid `AuditRecord.Decision` string value (no Go type change; `Decision` is already `string`). Rationale: keeps the change additive/non-breaking; the corpus emitter (Phase 23) sets `"alert"` for the Sentry detection-only surface.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| catalog source → policy engine | Untrusted catalog match data crosses into corroboration counting; CorroborateOutcome must not over-state source_count/tier |
| audit record → corpus store (Phase 23) | Attacker-influenced strings (Reason, command, paths) must be redacted before any corpus write; the redaction entrypoint is defined here |
| config file → Config struct | Operator-supplied corpus block is parsed; must not change decision behavior in v1 (Enabled defaults false) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Tampering (data integrity) | CorroborateOutcome tier mapping | mitigate | Tier derived from `count >= t.BlockAt`, never `level=="block"`; a single-source critical block yields `confidence_tier:"watch"` so the 2FA invariant is not falsified for downstream consumers (Pitfall 4) |
| T-22-02 | Information Disclosure | RedactRecordWithDefaults | mitigate | Cross-package redaction entrypoint exists so Phase 23's corpus store applies default credential redaction before writing; signature exposes no unexported type so the redaction call cannot be silently skipped |
| T-22-03 | Tampering | AuditRecord additive fields | accept | New fields are additive `omitempty`; FromDecision is unchanged; no existing decision behavior is altered (verified by the unchanged audit suite). Low risk: type-only schema slot. |
| T-22-04 | Elevation | config.CorpusConfig | mitigate | `Enabled` defaults false; the config block defines the type only and changes no decision surface in Phase 22 (store wiring is Phase 23) |
| T-22-SC | Tampering | npm/pip/cargo installs | accept | Phase 22 installs ZERO new dependencies (research §Package Legitimacy Audit); `go mod tidy` must leave go.mod unchanged — no supply-chain surface added |
