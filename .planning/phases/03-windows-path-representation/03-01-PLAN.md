<objective>
Fix the two confirmed WPATH-01 defect sites in the **Pollen** emitter (repo
`github.com/bantuson/pollen`, working tree `../pollen`, HEAD c94b271) so that `project_path`
in emitted NDJSON records uses native Windows backslash separators on Windows, while remaining
byte-identical on Linux/macOS.

Both `npm.IsNodeModulesPackageJSON` and `pnpm.IsPnpmStorePackageJSON` build a `projectPath` by
`strings.Join(..., "/")` over slash-split path segments. That forward-slash string leaks into the
`project_path` field of emitted records on Windows (e.g. `C:/Users/fana/code/web-app`). The fix
wraps each join in `filepath.FromSlash(...)`, which restores OS-native separators on Windows and
is a no-op on Unix.

Purpose: Satisfy WPATH-01 (native Windows path preservation, no Unix-to-Windows conversion
artifacts) without touching `internal/output/` and without drifting the Linux/macOS differential
test (D-04).
Output: Minimal-diff edits to npm.go (~line 114) and pnpm.go (~line 87), each accompanied by a
Windows-gated unit test asserting the returned projectPath uses backslashes + a drive letter.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → Pollen scanner | Path strings entering the npm/pnpm helpers originate from `filepath.WalkDir` over the scanned tree; they are filesystem-derived, not from a network or an agent prompt |
| Pollen emitter → NDJSON consumer | project_path/source_file strings flow out into NDJSON consumed downstream (beekeeper, audit log) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | project_path emitted from npm/pnpm helpers | accept | Path is reconstructed by `filepath.FromSlash` over segments derived from a path Pollen itself walked via `filepath.WalkDir`. No user/network input is parsed into the path; the fix changes only the separator representation, introduces no new parsing, and adds no new attack surface. |
| T-03-02 | Information Disclosure | project_path in NDJSON output | accept | project_path reveals on-disk layout, which was already emitted pre-fix (only the separator changes). This is in-scope expected behavior of an inventory scanner; no new field and no new disclosure beyond the existing path emission. |
| T-03-03 | Tampering | normalize_diff.go differential carve-out | mitigate | Do NOT edit normalize_diff.go (LOCKED). Pitfall 2: stripping path fields to "pass" the differential would hide the defect. The fix keeps Unix bytes identical via filepath.FromSlash so no normalization change is needed. Acceptance criteria forbid any normalize_diff.go edit. |

No HIGH-severity items. This is a separator-representation correctness fix over scanner-derived paths with no new untrusted-input path.
