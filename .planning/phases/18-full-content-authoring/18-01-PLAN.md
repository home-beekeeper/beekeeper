<threat_model>
## Trust Boundaries

This phase documents the shipped binary; it does NOT change the product. The only meaningful surface is **documentation that misleads a reader into a false-trust security decision**.

| Boundary | Description |
|----------|-------------|
| docs author → reader | A reader trusts a docs claim to make a security decision (which harness to use, whether a feature is enforced) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Information Disclosure (false trust) | accuracy_spec.py AC-2 | mitigate | The unenforced-label gate is built here; AC-2 fails any file mentioning release_age / lifecycle_script_allowlist without an "unenforced" label or `<UnenforcedCallout` — prevents shipping a config example that implies enforcement |
| T-18-02 | Spoofing (phantom command) | accuracy_spec.py AC-3 | mitigate | AC-3 fails any doc referencing a non-existent subcommand (`hooks status`, `catalogs rebuild`, `check --input`, `status`) — a phantom command sends a reader down a dead path and erodes trust |
| T-18-03 | Tampering (broken source trace) | accuracy_spec.py AC-1 | mitigate | AC-1 requires every Go-derived MDX to carry `source_doc:` AND that the referenced path exists — keeps claims anchored to a real source file (D-02) |
| T-18-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this phase. The component reuses already-installed `lucide-react`; `accuracy_spec.py` is stdlib-only. No legitimacy gate required. |
