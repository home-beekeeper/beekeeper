# Plan 29-01 — Per-rule posture severity (IPOVR-03, maintainer-directed at Gate 1)

## Goal

Let a user opt an individual posture rule UP from the default warn to block, via
layered config. A definite violation under block mode blocks the install; the
unknown path (missing timestamp / registry error / timeout) STAYS fail-soft warn
regardless of the configured action, so a registry outage still cannot break a
build. Untrusted (project) layers may only TIGHTEN (warn->block), never loosen.

## Design (Gate-1 load-bearing)

- Default stays warn + fail-soft (Phase 27, ratified). This plan only adds the
  opt-UP capability; it does not change the default.
- Unknown-stays-warn: block mode applies ONLY to a DEFINITE violation (release age
  below threshold, lifecycle scripts present, remote source present). On unknown
  inputs the rule warns, never blocks, even in block mode. Document this clearly;
  it is the line that keeps fail-soft intact.
- Tighten-only untrusted merge: mirror `failModeStrictness` / FailMode (layered.go
  TM-D-01). An untrusted layer can raise a rule warn->block; a block->warn (or
  ->off) from an untrusted layer is IGNORED. This is the IPOVR-03 self-defense.

## Tasks

### Task 1 — PostureConfig (internal/config)
Mirror the AutoQuarantine/CatalogSync sub-config pattern:
- `type PostureRuleConfig struct { Action string \`json:"action,omitempty"\` }` where
  Action is "" (treated as warn) | "warn" | "block".
- `type PostureConfig struct { ReleaseAge, Lifecycle, RemoteSource PostureRuleConfig \`json:"..."\` }`
  (the Allow list is added in Plan 29-02).
- `DefaultPostureConfig()` returns all-warn. `ValidatePostureConfig()` fail-closed:
  reject any Action not in {"", "warn", "block"}.
- Add `Posture *PostureConfig \`json:"posture,omitempty"\`` to Config (nil = defaults).
- Accessors: `(c Config) PostureRuleAction(rule string) string` returning the
  effective action ("warn" default) for "release-age"/"lifecycle"/"git-remote",
  nil-safe.
- Wire ValidatePostureConfig into Load (like ValidateAutoQuarantineConfig) so an
  invalid action is rejected fail-closed at load time.

### Task 2 — layered merge (tighten-only)
In `internal/config/layered.go`, mirror mergeLlamaFirewall / mergeAudit:
- `postureActionStrictness(action string) int`: warn(or "")=0, block=1.
- `mergePosture(dst, src)` (TRUSTED): src action wins when set.
- `mergePostureUntrusted(dst, src, layerName)` (UNTRUSTED): per rule, apply src
  action ONLY when `postureActionStrictness(src) >= postureActionStrictness(dst)`
  (tighten-only); a looser src action is ignored (log like the other untrusted
  levers). Wire both into merge() and mergeUntrusted(). Add the Posture root-default
  guard in LoadLayered if needed (mirror mergeCorpus/mergeAutoQuarantine — the
  v1.4.0 FRB-05 bug was a missing merge; do NOT repeat it: PostureConfig MUST be
  merged in BOTH merge paths).

### Task 3 — adapter applies the action
In `internal/check/posture_adapter.go`, make `posturize` action-aware:
- Replace `posturize(d)` with `posturizeWithAction(d policy.Decision, action string)`:
  if the rule did not fire (allow) -> unchanged; if it FIRED and action=="block" ->
  return the BLOCK decision (Allow:false, Level:"block", keep Reason+RuleIDs);
  else -> warn (current behavior).
- `warnUnknown` (the fail-soft unknown path) is UNCHANGED: it always warns,
  regardless of action. Add a comment: block mode never blocks an unknown.
- Thread `cfg.PostureRuleAction(...)` for each rule (release-age / lifecycle /
  git-remote) into the per-rule posturize call. `evaluatePosture` already receives
  `cfg config.Config`; use it (the `_ = cfg` line goes away).
- Most-restrictive merge in handler.go is unchanged: a posture block now CAN block
  (when opted up), and a catalog/sensitive-path block still wins; posture can still
  never DOWNGRADE another block.

### Task 4 — tests
- Config: ValidatePostureConfig rejects a bogus action; DefaultPostureConfig all-warn;
  accessor nil-safe + returns the configured action.
- Layered: an untrusted (project) layer raising release-age warn->block IS applied;
  an untrusted layer lowering block->warn is IGNORED (the tighten-only invariant,
  a TM-D-style test alongside the existing FailMode/LlamaFirewall ones).
- Adapter / live path (extend posture_adapter_test + posture_integration_test):
  with PostureConfig release-age action=block, a <24h package BLOCKS at the hook
  (exit non-zero) while a missing-timestamp still WARNS (unknown stays fail-soft)
  even under block mode; default (no Posture config) still warns. A catalog block
  still wins over a posture block (both block; assert block + the catalog reason).
