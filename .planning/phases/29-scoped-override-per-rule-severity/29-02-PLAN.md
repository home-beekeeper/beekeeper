# Plan 29-02 — Scoped override for posture decisions

## Goal

When posture warns or blocks an install, offer graduated, scoped, audited
responses instead of a binary on/off: allow once, allow always (with a recorded
reason), or block (the default stands). Each writes a distinct audit-log entry.
Allow-always persists as a scoped standing exception; allow-once is a one-time
exception consumed on the next matching install.

## CRITICAL security distinction (do not get this wrong)

Posture allow-always must use a POSTURE-SCOPED allowlist, NOT the general
`package_allowlist` policy rule. `ApplyPolicyOverlay` (policyloader/enforce.go)
treats a `package_allowlist` "allow" as a user-trust override that downgrades a
CATALOG-corroborated block (T-09-31, THREAT-MODEL §1). If posture allow-always
reused `package_allowlist`, silencing a fresh-package posture WARN would also
whitelist that package past MALWARE detection. So posture allow-always lives in a
posture-scoped allowlist that the posture adapter consults and that NEVER touches
catalog/corroboration enforcement.

## Design

- **Allow-always** = a `PostureAllow{Ecosystem, Package, Rule, Reason}` entry
  (Rule="" = all posture rules) appended to `PostureConfig.Allow` in the USER
  config layer. Untrusted (project) layers may NOT add Allow entries (adding an
  allow loosens; tighten-only). The posture adapter passes matching allows as the
  `Exclude` allowlist to each pure evaluator (EvaluateReleaseAge/Lifecycle/
  RemoteSource already accept an exclude/allowlist), so an allowlisted package
  stops firing posture rules but catalog enforcement is untouched.
- **Allow-once** = a one-shot token in an owner-only, atomically-written store
  (e.g. `<stateDir>/posture-allow-once.json`, mirror the corpus salt file's
  owner-only/atomic pattern). The posture adapter checks the store; a matching
  (ecosystem,package) token -> allow AND consume the token (atomic rewrite without
  it). A consume is the only write the adapter makes, and only on a match.
- **Block** = the default action stands; no override is written (a no-op choice
  that is still audited as the operator's recorded decision if invoked explicitly).

## Tasks

### Task 1 — posture-scoped allowlist (config + adapter)
- Add `Allow []PostureAllow` to `PostureConfig` (29-01) with the type above. Layered
  merge: Allow entries flow from TRUSTED layers only; mergePostureUntrusted drops
  src.Allow (loosening). ValidatePostureConfig validates Rule in {"", "release-age",
  "lifecycle", "git-remote"}.
- `evaluatePosture` builds the per-rule Exclude list from `cfg.Posture.Allow`
  (entries whose Rule is "" or matches the rule, and whose Ecosystem matches or is
  empty) and passes it to each evaluator's config (ReleaseAgeConfig.Exclude,
  RemoteSourceConfig.Exclude, the lifecycle allowlist). An allowlisted package ->
  allow for that rule. Catalog path is NOT consulted here, so malware enforcement
  is unaffected.

### Task 2 — allow-once store
- New `internal/check/posture_allowonce.go` (or internal/posture): owner-only,
  atomic JSON store of one-shot {ecosystem, package} tokens with the recorded
  reason + timestamp. `AddAllowOnce`, `ConsumeAllowOnce(eco, pkg) bool` (returns
  true and removes the token if present). The adapter calls ConsumeAllowOnce BEFORE
  firing the rules; a hit -> the whole posture eval is allow for this install.
- Bounded (cap entries), fail-open on a read error (a corrupt one-shot store must
  not block — it is an override convenience, not a security gate), but a successful
  consume must be durable (atomic write).

### Task 3 — the override CLI (cmd/beekeeper)
Under `beekeeper posture`:
- `beekeeper posture allow <pkg> [--ecosystem E] [--rule release-age|lifecycle|git-remote] (--once | --always) --reason "..."`:
  - `--always`: append a PostureAllow to the user config (load -> append -> save,
    reusing the config write + audit plumbing) and write an audit record.
  - `--once`: AddAllowOnce(eco, pkg, reason) and write an audit record.
  - `--reason` is REQUIRED for --always (a recorded reason; PRD). 
- `beekeeper posture enforce <rule> (--block | --warn)`: set
  `PostureConfig.<rule>.Action` in the user config (this is the IPOVR-03 opt-up CLI
  surface) + audit record. `--warn` lowers it back (a TRUSTED user layer may lower
  its own setting; only UNTRUSTED layers are tighten-only).
- Each override writes a DISTINCT audit record: record_type `posture_override`,
  with a field for the action (`allow_once` | `allow_always` | `enforce_block` |
  `enforce_warn`), the package/rule, and the reason. Redact the reason like other
  audit fields.

### Task 4 — TUI incident card (display-only)
In the dashboard, when a posture warn/block appears in the activity/alerts feed,
the incident card shows the three graduated options and the EXACT `beekeeper
posture ...` command to enact each. Display-only: the TUI does not write the
override itself (the dashboard stays read-only by default; the CLI is the
actionable surface). Mirror an existing read-only card.

### Task 5 — tests
- allow-always: a PostureAllow for pkg X makes a <24h X install ALLOW (no posture
  warn) while a non-allowlisted fresh package still warns; an allow-always entry
  does NOT suppress a CATALOG block for X (the security distinction — assert a
  catalog-blocked allowlisted package still blocks).
- allow-once: a one-shot token makes the next X install allow, and the SUBSEQUENT X
  install warns again (token consumed); the store is owner-only.
- CLI audit: each of `allow --once`, `allow --always`, `enforce --block` writes the
  correct distinct `posture_override` audit record with the reason (IPOVR-02). Drive
  via the live audit path, not just the record builder.
- Untrusted layer cannot add an Allow entry (loosening dropped) — a project-config
  PostureAllow is ignored.
