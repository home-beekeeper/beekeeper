<objective>
Build the differential test harness (PTEST-02) — the single most load-bearing guard in this milestone. Raw NDJSON comparison between `pollen` and upstream `bumblebee` ALWAYS fails because upstream emits 7 non-deterministic fields and orders package records by concurrent-worker completion. This plan implements the normalization harness (strip the 7 fields, sort by the stable `record_id` SHA-256 key) and the `TestDifferential` that runs both binaries on a fixed fixture and asserts byte-for-byte equality AFTER normalization. It also adds the selftest regression assertion (PTEST-03) so the 3-finding contract is locked.

Purpose: Every later Windows-code phase (2–4) relies on this guard to prove Pollen has not drifted from upstream behavior on Linux/macOS. If this harness is wrong, all downstream parity claims are worthless. TDD is mandatory here: write the normalization tests first (known input → known stripped+sorted output), then implement.

Output: `normalize_diff.go` (normalization), `differential_test.go` (`TestDifferential` + selftest regression), and a fixed `testdata/diff-fixture/` — committed to the pollen repo, green on the dev host.

<normalization_contract>
<!-- LOAD-BEARING. From 01-RESEARCH.md "NDJSON Determinism Analysis" + 01-VALIDATION.md. Verified against upstream model.go / endpoint.go / main.go / scanner.go. -->
Fields to STRIP before comparison (7 total):
  per-scan (4):  run_id, scan_time, end_time, duration_ms
  per-machine endpoint sub-fields (3):  endpoint.hostname, endpoint.username, endpoint.uid
Sort key after stripping:  record_id  (upstream's StableID() — SHA-256 of record_type+ecosystem+package_name+version+source paths; same input => same hash). This is the safe, deterministic sort anchor.
Deterministic fields that MUST match (do NOT strip): record_id, record_type, schema_version (always "0.1.0"), ecosystem, package_name, version, source_file, finding fields (catalog_id, severity).

Why ordering is non-deterministic: upstream scanner spawns 4 goroutines over a 256-item buffered channel; package records emit in worker-completion order, not filesystem order. Two runs on identical input differ in record order. Sorting by record_id removes this.

Environment control for the fixture run (so root discovery + endpoint do not vary): pass an explicit `--root <fixture-dir>` (do NOT let it discover a dynamic home dir); use a profile that honors explicit root (deep/project per upstream flag semantics). If upstream exposes a device-id env override flag, pin it to a fixed value so endpoint.device_id is stable too (or strip it if it is non-deterministic in practice — verify against actual output).
</normalization_contract>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| upstream binary build (in-test clone) → differential assertion | The test fetches upstream code over the network to build the comparison oracle. |
| pollen output behavior → downstream parity claims | This harness is the evidence every later phase cites to prove no behavioral drift. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Tampering | The in-test upstream clone could pull a moving/compromised newer commit, making the oracle wrong | mitigate | The test checks out the explicit pinned SHA `c24089804ee66ece4bec6f14638cb98985389cdb` (asserted by grep gate), never a branch/tag-latest. |
| T-01-10 | Repudiation | A silently-passing differential test would mask real drift (false assurance) | mitigate | All skip paths (Windows, offline-clone, missing record_id) use structured `t.Skip`/error with explicit reasons; a normalize() failure surfaces a diff. Never a silent pass. |
| T-01-11 | Tampering | Hidden non-deterministic field added by a future upstream sync would falsely fail (or falsely pass) the diff | accept | The map-based normalize tolerates additive fields; if upstream adds a NEW non-deterministic field in a Phase-5 sync, the differential test will fail loudly and the strip-list is updated then — a deliberate fail-loud, not a Phase-1 risk. |
</threat_model>
