<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Windows env vars (%APPDATA%/%LOCALAPPDATA%/%USERPROFILE%/%ProgramFiles%) → root construction | Env vars are process-owner-controlled inputs read via os.Getenv; an empty or attacker-set value feeds filepath.Join |
| Candidate root paths → filesystem (os.Stat / glob) | Constructed paths are stat-ed/globbed by globExisting and filterExistingRoots; symlinks/junctions may redirect |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering / Elevation | `windowsBaselinePackageRoots` env-var reads | mitigate | Guard every os.Getenv at the env-var level (`if appdata := os.Getenv("APPDATA"); appdata != ""`) so an empty var never yields a CWD-relative path via `filepath.Join("", ...)` — RESEARCH Q5 Pitfall 1. Verified by Plan 02 empty-env test. |
| T-02-02 | Information Disclosure | Junction/symlink traversal under %LOCALAPPDATA% | accept | Bounded risk: at worst a package is double-counted; record_id is a content SHA so duplicates dedupe downstream. Documented Open Research Flag (CONTEXT). No new traversal surface beyond existing Unix glob/stat behavior. |
| T-02-03 | Denial of Service | Absent/hostile candidate root path | mitigate | Reuse existing `filterExistingRoots` (os.Stat is-dir filter) — absent roots are silently dropped, never errored, preserving the upstream fail-safe. No new "missing root" code path invented. |
</threat_model>
