<objective>
Implement Layer 2 of the editor-extension defense: the `beekeeper watch` file-watcher daemon (`internal/watch/`) plus the best-effort desktop-notification wrapper (`internal/notify/`). This is the core new technical surface of Phase 3.

Purpose: GUI installs and editor auto-updates bypass the Layer 1 CLI intercept entirely — this is exactly the Nx Console vector. The watcher detects a new extension directory via OS-native filesystem events and runs it through the same Phase 2 catalog + release-age machinery within seconds, then quarantines + notifies on a hit. The CLI wiring of `beekeeper watch` lives in plan 04; this plan delivers the daemon logic and handler as testable library code.

Output: `Watch()` daemon loop, `ParseManifest()`, `HandleNewExtension()` orchestrating catalog match + release-age + audit + notify + quarantine, and `notify.Notify()` best-effort wrapper.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem event → handler | An untrusted, possibly attacker-created directory + package.json crosses into the handler |
| remote registry → handler | Untrusted marketplace timestamp crosses into the release-age decision |
| handler → desktop session | Notification content (publisher/name) crosses into the OS notification daemon |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | Windows NTFS duplicate Write events causing double-quarantine | mitigate | shouldProcess filters to Create-only on Windows (runtime.GOOS); 500ms debounce coalesces bursts; quarantine.Move on an already-moved dir errors and is logged, not panicked. |
| T-03-03-02 | Tampering | fsnotify event injection via symlink pointing outside a watched root | mitigate | Handler holds WatchedRoots and verifies filepath.Dir(Clean(path)) is a configured root before processing; events outside roots are rejected. |
| T-03-03-03 | Denial of Service | Notification spam when 100 extensions are installed rapidly | mitigate | Debounce → one notification per extension; notifications are best-effort and dropped on failure, never queued. |
| T-03-03-04 | Tampering | Malicious package.json with path-traversal publisher/name | mitigate | Identity is parsed and validated (empty publisher/name → ErrNoManifest); path construction for quarantine is delegated to quarantine.Move which applies filepath.Base + prefix guard (T-03-02-01). |
| T-03-03-05 | Information Disclosure | beeep notification leaking on a headless/SSH session error | accept | Notification failure is swallowed; on Linux a DISPLAY/WAYLAND check short-circuits. No security impact — quarantine still proceeds. |
| T-03-03-06 | Repudiation | Quarantine action without an audit trail | mitigate | Every HIT writes a critical sentry_alert record BEFORE the notification; audit-write failure is logged but the quarantine still proceeds (fail-closed on the security action). |
</threat_model>
